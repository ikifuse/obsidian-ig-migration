# 仕様工程ワークフロー

## この工程の役割

- 仕様は、設計で確定した内容を、実装者が未確定部分を推測せずコード化できる形式へ変換する工程である。
- workflowは仕様化の手順だけを定め、仕様本文の実装内容を代わりに定義しない。

## 開始条件と対象

- オーナーが設計を確認し、仕様工程への移行を承認していることを開始条件とする。
- `04_IG移行仕様書/`の復旧済みv1.2仕様書は比較基準であり、分割後設計へ対応済みとは推測しない。
- 今回の対象仕様書と、対応する設計分冊だけを読む。コード・成果物は比較に必要な対象範囲だけを読む。

## 分割後設計との対応確認

- 設計書の分割だけを理由に仕様内容を変更しない。
- 設計参照先の不一致、現在も有効な仕様、設計変更候補、未確認事項を区別する。
- 要約、言い換え、翻訳だけを仕様変更の根拠にしない。
- 期待に近い現行コード・成果物がある範囲では、成立している部分を維持し、全面再仕様化を避け、承認された最小差分を優先する。

## 仕様化

1. 対象となる設計判断と、今回仕様化する範囲を明示する。
2. 設計上の意味、責任境界、例外、禁止事項、成立条件を維持する。
3. 処理の流れに沿って、入出力、項目名、データ型、初期値、ファイル名、保存先、処理順、判定式、アルゴリズム、画面操作を必要な範囲で確定する。
4. 曖昧な表現を構造化し、例外を含む処理手順を実装可能な粒度にする。
5. 未確定部分を推測で補わず、未確認事項として分離する。

設計の矛盾、欠落、論理的・物理的な不成立を発見した場合は仕様化を停止し、設計工程へ戻す。一時的なIssue、未承認の発見、将来構想を現在の実装要件へ変換しない。

## 完了条件

- 変更した仕様、未変更範囲、コード・成果物への影響、未確定事項を報告する。
- 現在の進捗、完成主張、次の作業を仕様本文へ混ぜない。
- 仕様変更の承認を、コード変更または成果物再生成の承認へ拡大しない。
- オーナーが仕様を確認し、実装工程への移行を承認するまでコードを変更しない。

# 恒久ルールの追加・修正基準

この文書は、プロジェクト運用ルールを「追加」「既存文の修正」「文書化しない」のどれにするか判断するときだけ使う。企画・設計・仕様の内容やコマンド権限は決めない。

## 判断手順

1. 同じ誤り・誤解・修正指摘が複数回あり、将来も必要なプロジェクト固有ルールか確認する。
2. AGENTS.mdまたは対応workflowに同じ責任の規則がないか確認する。
3. 原因が既存文の曖昧さなら、新規追加せず既存文を直す。
4. コード、テスト、lint、hook、自動化で防げるなら機械的対策を優先し、人間の判断境界だけを文書に残す。
5. 一時的な進捗・引継ぎ・未確認事項は恒久ルールにしない。例やスクリプトを伴う反復作業はSkill化も検討する。
6. 変更対象、理由、影響を示し、オーナー承認後に反映する。

## 記述条件

- 適用対象、対象外、条件、例外、承認範囲を、意味が変わらない最小限の文で記す。
- 確認済み事実、仮定、提案を区別する。
- 要約や翻訳だけを変更根拠にせず、公式資料、承認済み文書、Git履歴と照合する。
- 根拠を確定できない場合は変更せず、未確認事項として報告する。

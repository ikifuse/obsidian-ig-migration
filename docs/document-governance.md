# 文書・資料の配置と退役

この文書は、資料の置き場所または退役方法を決めるときだけ使う。企画・設計・仕様の内容や承認条件は決めない。

## 配置

| 対象 | 配置と条件 |
|---|---|
| 実データ解析、未完了の調査、未確定の判断、現役の引継ぎ | `03_継続中の判断材料・引継ぎ`。個人情報を含む原本と詳細はローカルに保ち、公開承認された資料だけをGit管理する |
| 現役の引継ぎ書 | 同じ作業の流れなら既存書を更新する。作業の流れ、判断範囲、読む相手が変わる場合だけ新規作成する |
| 再利用する補助ツール | `tools/` |
| 一回限り・役割を終えた補助ツール | `99_完了済み参考資料` |
| 役割を終えた文書・置き換え済み成果物 | `99_完了済み参考資料` |

- 公開資料が必要な場合は原本を変更せず、個人情報を除いた版を別に作る。
- GitHub Issuesは、オーナーが指定したときだけ公開可能な引継ぎ、ブロッカー、短期状態に使う。非公開の詳細はローカル資料へ分離する。
- 補助ツールが独立した目的と継続的な役割を持つ場合は、このリポジトリに残すか別リポジトリへ分けるかをオーナーが判断する。
- 新しいフォルダー、成果物群、補助ツールには、目的と現役・退役の状態が分かる短い説明を置く。

## 分割前設計書

- `03_継続中の判断材料・引継ぎ/02_IG移行設計書.md`は、分割後設計の復元・監査と、旧仕様との未完了の対応確認に使う継続中資料とする。
- 通常の設計判断では読まない。旧仕様との対応確認完了後の`99_完了済み参考資料`への移動も自動で行わず、オーナーが別に判断する。

## 退役

1. 対象、退役理由、関連資料への影響を確認する。
2. 原則として削除せず`99_完了済み参考資料`へ移し、置き換え先または役割を終えた理由を記録する。
3. 現役文書には承認済みの現行内容だけを残し、旧内容はGit履歴または`99_完了済み参考資料`で追跡可能にする。
4. 引継ぎ書や参考資料へ現役文書全文を複製せず、参照先、置き換え理由、未確認事項だけを残す。

削除は、オーナーが対象とmacOSの通常のゴミ箱への削除を明示した場合に限る。その指示を関連資料や同じフォルダーへ拡大しない。完全削除には別の明示指示が必要である。

# 企画工程ワークフロー

## この工程の役割

- 現役企画書は、プロジェクトで「何を、なぜ実現するか」と、承認済みの対象範囲・到達点を定める最上位文書である。
- 正本は`01_IG移行企画書v1.0.md`とする。このworkflowは企画書の扱い方だけを定め、本文を置き換えない。

## 現役企画と仮メモ

現役企画では、次を承認済み条件として扱う。

- 目的、理由、実現したい価値、設計思想
- 対象範囲と対象外
- 長期的な最終到達点と、現在承認されている完走点
- 人間が最終判断を行う原則

思いつき、仮案、未検証の制約、未確定の判断は、現役企画へ直接混ぜず`03_継続中の判断材料・引継ぎ`に置く。文章未整理や矛盾を含んでもよいが、オーナーが統合を承認するまで企画条件にしない。

現役企画書の全体を未確定または下書きと解釈しない。長期的な最終到達点と現在承認されている完走点も区別する。

## 成立性に問題が見つかった場合

1. 仕様または実装で見つかった問題も、まず設計工程へ戻して成立性を検証する。
2. 企画目的を維持した複数の実現方法と回避策を比較する。
3. 技術的難度、作業量、AIのコンテキスト不足だけを理由に企画を縮小しない。
4. 論理的または物理的に成立しない場合だけ、次を揃えて企画変更候補を提示する。
   - 成立しない企画条件
   - 確認済み事実と未確認事項
   - 検討した実現方法と、成立しない理由
   - 企画変更が設計・仕様・コード・成果物へ与える影響
5. オーナーの承認前に企画書を変更しない。

## 反映と完了

- 承認された企画変更は現役企画の該当箇所へ統合し、旧判断はGit履歴または`99_完了済み参考資料`で追跡可能にする。
- 企画書には、設計構造、仕様詳細、実装方法、進捗、引継ぎ、未承認の調査結果を混ぜない。
- 変更後の企画書をオーナーが確認し、設計工程への移行を承認した時点を、この工程の停止点とする。

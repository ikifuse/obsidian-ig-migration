# AGENTS

## 目的と最上位条件

- `01_IG移行企画書v1.0.md`を、プロジェクトの目的・理由・到達点を定める最上位条件とする。
- 現在承認されている工程を、既存成果を壊さず完走へ近づける。長期目的は判断基準であり、新機能追加や工程拡大の許可ではない。
- 文書とオーナーへの報告は、専門用語を必要最小限にした日本語で記述する。

## 全工程で守る判断原則

- 企画、設計、仕様、実装、成果物検証を混ぜない。上位工程の内容を下位工程の都合で変更しない。
- 確認済みの事実、仮定、提案、未確認事項を区別し、未確認の解釈を確定事項として扱わない。
- 上位文書とコード・成果物の不一致では、どちらかを自動的に正解としない。「維持」「微調整」「設計変更候補」「実装不具合」「未確認」に分類し、影響を示して変更方針の承認を得る。
- 期待に近い既存成果物がある範囲では、成立している部分を維持し、全面再設計・全面再仕様化を避け、承認された最小差分を優先する。
- 全体完走を判断基準にするが、それを指定範囲外の変更許可とは解釈しない。

## 作業範囲と停止条件

- 作業開始時に、対象、操作、停止点を明示する。「進めてください」は、直前に合意した対象と停止点にだけ適用する。
- 閲覧許可、リポジトリ全体へのアクセス許可、完走の依頼は、変更許可ではない。承認されたファイルだけを変更する。
- 指定範囲外で問題を見つけた場合は、事実と影響を報告し、変更しない。範囲外の変更が不可欠なら、理由、対象、予定変更、影響を示して追加承認を得る。
- 成果物が変わる複数の合理的な解釈がある場合は、推測で選ばず確認する。誤操作の巻き戻しでも、別ファイルへ波及する場合は同様とする。
- 前工程を更新しても、オーナーが確認して次工程への移行を承認するまで進まない。停止点では、結果と未変更範囲を報告して停止する。

## 読み込みルーター

リポジトリ全体を最初から一括して読まず、現在の工程の行だけを適用する。追加資料は、その資料が今回の判断に必要な理由を特定できる場合だけ読む。

| 工程 | 最初に読むもの | 必要な範囲だけ追加で読むもの |
|---|---|---|
| 企画 | `docs/planning-workflow.md`、`01_IG移行企画書v1.0.md`の対象箇所 | 企画判断の根拠となる資料 |
| 設計 | `docs/design-workflow.md`、`02_IG移行設計書/00_設計書目次.md` | 目次から選んだ対象分冊。複数分冊が必要なら理由を示す |
| 仕様 | `docs/specification-workflow.md`、対象仕様書、目次から選んだ対応設計分冊 | 比較に必要な対象コード・成果物 |
| 実装・成果物検証 | 対象仕様書、対象コード、対象成果物 | 不一致の判断に必要な対応設計分冊 |
| 文書の配置・退役 | `docs/document-governance.md` | 配置判断に必要な対象資料 |
| 恒久ルールの追加・修正 | `docs/rule-addition-criteria.md` | 根拠となる公式資料・承認済み文書・履歴 |

- 設計の正本は`02_IG移行設計書/00_設計書目次.md`と、そこに記載された設計6冊とする。
- 分割前の`03_継続中の判断材料・引継ぎ/02_IG移行設計書.md`は復元・監査用であり、原文移送監査、復元確認、旧仕様との対応確認を依頼された場合だけ読む。現在の設計判断には使わない。
- 仕様書は`04_IG移行仕様書/`に置く。復旧済みv1.2が分割後設計へ対応済みとは推測しない。
- `docs/`は工程の進め方を定める。企画書、設計書、仕様書の本文正本ではない。
- `09_IGC統合/`配下の作業では、本文書に加えて`09_IGC統合/AGENTS.md`を適用する。工程間の境界とプロジェクト共通原則は親側の正本が定める。
- `10_Memory_Synapse_DB/`配下の作業では、本文書に加えて`10_Memory_Synapse_DB/AGENTS.md`を適用する。工程間の境界とプロジェクト共通原則は親側の正本が定める。

## 対象環境とデータ保護

- 最終動作環境はmacOS / iMac M4とし、macOSの権限、パス、ターミナル環境を含めて検証する。
- 操作はiMac内部ストレージのローカル作業フォルダーを優先する。明示された範囲を除き、クラウド同期フォルダー、外部ドライブ、外部ネットワーク、システム領域、無関係なパスへアクセスしない。
- 大量データの一括処理は、対象、操作、影響を示して事前承認を得る。
- 実データ解析、未確定の調査、作業中の引継ぎは`03_継続中の判断材料・引継ぎ`へ置く。個人情報を含む可能性がある資料は、公開承認されたもの以外Git管理しない。

## 編集・記録・外部操作

- 編集前に差分を確認し、既存のユーザー変更と今回の変更を分ける。編集後は、変更ファイル、未変更範囲、検証結果、未確定事項を報告する。
- 承認済みの判断は現役本文の該当箇所へ統合し、新旧の判断を末尾へ積み重ねない。進捗、完成主張、次の作業を設計書・仕様書へ混ぜない。
- 設計変更が実装へ影響する場合も、設計承認、仕様変更承認、コード変更承認を工程ごとに分ける。
- 文書構造変更の承認を、仕様書、コード、成果物の変更承認へ拡大しない。
- 設計内容の変更、仕様変更、コード変更、成果物再生成は、それぞれ対象と影響を示して別途承認を得る。
- Git履歴だけに主要な判断を依存させず、オーナーがGit操作なしで現状と判断経緯を確認できる状態を保つ。
- README、GitHub Issues、ステージ、コミット、pushは、オーナーが対象操作を明示した場合だけ行う。
- 役割を終えた資料の扱いは`docs/document-governance.md`に従う。明示された対象以外を削除・移動しない。

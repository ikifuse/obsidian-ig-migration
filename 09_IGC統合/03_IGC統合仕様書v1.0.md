# 03_IGC統合仕様書v1.0

## 1. 目的

本仕様書は、[[02_IGC統合設計書v1.0]]と、分割後の正式設計書で確定したIGCの責任に基づき、IGP・IGR・IGSのSynapseカードとSystemLogs一覧を`output_IGC`へ統合する具体的な規則を定義する。

IGCが行うのは、同一カテゴリ・完全同一表記のカードを一件へ整理し、関連投稿を集め、カードと一覧を対応させる処理だけである。表記が異なるカードの意味融合、カテゴリ横断融合、大きなメモ、共通作業台、融合・分離・判断履歴は扱わない。

## 2. 実行範囲

### 2.1 入力系統

入力系統と処理順を次に固定する。

1. `output_IGP`
2. `output_IGR`
3. `output_IGS`

カテゴリごとの入力は次のとおりとする。

| カテゴリキー | Synapseカード | SystemLogs一覧 |
|---|---|---|
| `tags` | 各系統の`Instagram_Logs/Synapses/Tags/*.md` | 各系統の`Instagram_Logs/SystemLogs/ハッシュタグ一覧.md` |
| `mentions` | 各系統の`Instagram_Logs/Synapses/Mentions/*.md` | 各系統の`Instagram_Logs/SystemLogs/メンション一覧.md` |
| `locations` | 各系統の`Instagram_Logs/Synapses/Locations/*.md` | 各系統の`Instagram_Logs/SystemLogs/場所一覧.md` |

入力ディレクトリと一覧ファイルは、カードまたは一覧項目が0件の場合も存在しなければならない。0件のLocations等は正常な空入力として扱う。

macOSがファイル名をUnicode正規化して保存する場合、固定された一覧ファイル名はNFCまたはNFDの正規化差だけを同じ物理名として探索してよい。同じ正規化名に複数ファイルが対応する場合は異常終了する。この扱いを、カードの人間向け表記の完全一致判定には適用しない。

### 2.2 恒久出力

恒久的に作成または置換する対象は次の6か所だけとする。

* `output_IGC/Instagram_Logs/Synapses/Tags`
* `output_IGC/Instagram_Logs/Synapses/Mentions`
* `output_IGC/Instagram_Logs/Synapses/Locations`
* `output_IGC/Instagram_Logs/SystemLogs/ハッシュタグ一覧.md`
* `output_IGC/Instagram_Logs/SystemLogs/メンション一覧.md`
* `output_IGC/Instagram_Logs/SystemLogs/場所一覧.md`

### 2.3 対象外

次は入力、生成、変更、削除、補完の対象外とする。

* `output_IGX`
* Posts
* Reels
* Stories
* media
* Timeline
* RawData
* 実際のObsidian Vault
* iCloud配下

既存の`output_IGC`は、入力、正解、補完元、差分吸収元として使用しない。

## 3. Synapseカードの入力形式

### 3.1 文字コードと改行

カードと一覧はUTF-8として読み取る。デコードできない場合は異常終了する。

改行コードは読み取り時にLFへ統一して比較する。Unicode文字、空白、インデント、コメント、大文字小文字、全角半角は正規化しない。

### 3.2 完全一致に使用する人間向け表記

物理ファイル名は統合キーに使用しない。カード内の次の値を、人間向け表記として読み取る。

| カテゴリ | 必須の値 | 例 |
|---|---|---|
| Tags | `hashtag_note.hashtag` | `#SampleTagA` |
| Mentions | `mention_note.mention` | `@sample_account_001` |
| Locations | `location_note.location` | `大阪城` |

値は引用符を外した文字列として取得する。前後空白の削除、Unicode正規化、大文字小文字変換、記号変換、別名解決は行わない。

同じカードについて、次の四つが完全に同じ文字列でなければならない。

1. Frontmatterの`aliases`にある唯一の値
2. 本文先頭のH1見出し
3. 上表のカテゴリ固有値
4. 対応するSystemLogs項目のWikiリンク表示名

Frontmatterの`tags`は、カテゴリごとに`Synapse/Tag`、`Synapse/Mention`、`Synapse/Location`のいずれか一つと一致しなければならない。

必須値がない、複数ある、値の型が文字列ではない、四つの値が一致しない、カテゴリが一致しない場合は異常終了する。

### 3.3 カードの共通情報と関連投稿

カードには`## 関連投稿`見出しを一つだけ置く。この見出しより前を「共通情報」、見出しより後を「関連投稿領域」とする。

関連投稿領域は、空行と次の形式のWikiリンクだけを許可する。

```markdown
[[投稿ID]]
```

別名、見出し、ブロック参照、埋め込み、Markdown箇条書き、通常文章を関連投稿領域に置かない。投稿リンクが0件のカードは異常とする。

同じカード内に同一投稿リンクが複数ある場合は異常終了する。

### 3.4 同一入力系統内の一意性

一つの入力系統・一つのカテゴリ内で、同じ人間向け表記を持つカードは一枚だけでなければならない。

物理ファイル名が異なっていても、人間向け表記が同じカードが複数存在する場合は、どちらかを選ばず異常終了する。

## 4. SystemLogs一覧の入力形式

### 4.1 一覧項目

一覧項目は次の行で開始する。

```markdown
- [x] [[物理リンク先|人間向け表記]]
```

`- [X]`も採用状態として読み取る。入力側に`- [ ]`が一件でもある場合は、人間による除外判断が存在する可能性があるため、その項目を無視または自動除外せず、対象の入力系統、カテゴリ、人間向け表記を報告して全処理を停止する。

入力一覧の全項目が`- [x]`または`- [X]`である場合だけ、Synapseカードと一覧に存在する全項目を照合・統合する。

人間向け表記に`.md`を付けない。物理リンク先は、対応する入力Synapseカードの`.md`を除いた物理ファイル名と一致しなければならない。フォルダ付きリンクの場合は最後のパス要素を比較する。

同じ一覧内に同じ人間向け表記が複数ある場合は異常終了する。

### 4.2 出現回数と出現投稿

各項目は次の形式を持つ。

```markdown
出現回数: N回

出現投稿:
  - [[投稿ID]]
```

`N`は0以上の整数として解析し、出現投稿リンク数と一致しなければならない。同じ項目内に同一投稿リンクが複数ある場合は異常終了する。

出現投稿リンクが0件の項目は異常とする。

### 4.3 カードと一覧の入力照合

統合前に、入力系統・カテゴリごとに次をすべて確認する。

* カードの人間向け表記集合と一覧の人間向け表記集合が一致する。
* 一覧の物理リンク先が、同じ人間向け表記を持つ実カードを指す。
* カードの関連投稿リンク列と一覧の出現投稿リンク列が、件数、値、順序とも一致する。
* 一覧の出現回数が、カードの関連投稿リンク数と一致する。

一覧だけ、カードだけ、リンク先不一致、関連投稿不一致、件数不一致を自動修復しない。いずれか一つでも不一致があれば、統合出力を作らず異常終了する。

## 5. 完全一致統合

### 5.1 統合キー

統合キーは、次の二要素の組とする。

```text
(カテゴリキー, 人間向け表記)
```

人間向け表記はPython文字列の完全一致で比較する。大文字小文字、全角半角、Unicode正規化形、先頭記号、句読点その他の文字が一文字でも異なれば別の統合キーとする。

例えば`#ABC`と`#abc`、`#abc`と`＃abc`、`大阪城`と`大阪城天守閣`は別カードとして出力する。カテゴリが異なる同一文字列も別カードとする。

### 5.2 共通情報の一致

同じ統合キーが複数系統に存在する場合、3.3で分けた共通情報を比較する。

改行をLFへ統一した後の共通情報が文字単位で完全一致する場合だけ、一組を出力する。比較ではFrontmatter、見出し、カテゴリ固有YAML、人間が育てる欄、コメント、空白、その他の本文を省略しない。出力には、2.1の入力順で最初に存在するカードの共通情報を使用する。

共通情報が異なる場合は、最初の系統を採用せず、内容を混ぜず、全処理を異常終了する。

### 5.3 関連投稿の加算

同じ統合キーに属する関連投稿リンクを、次の順番で一つの列へ加える。

1. `output_IGP`カード内の順番
2. `output_IGR`カード内の順番
3. `output_IGS`カード内の順番

投稿リンクの意味、日時、画像、出来事を比較してまとめない。リンク文字列を変更しない。

同じ統合キーについて、複数系統に完全に同じ投稿リンクがある場合は、重複除去せず入力異常として全処理を停止する。カードと対応一覧に同じリンクが一回ずつ記載されることは正常な相互表現であり、重複とは数えない。

### 5.4 一系統だけに存在するカード

一系統だけに存在する統合キーも出力する。共通情報と関連投稿は同じ規則で再構成し、推測した情報や他系統の空欄を追加しない。

### 5.5 出力カード形式

出力カードは、共通情報を一組置き、その後に次の形式で統合後の関連投稿を置く。

```markdown
（入力と完全一致する共通情報）

## 関連投稿

[[IGPの投稿ID]]
[[IGRの投稿ID]]
[[IGSの投稿ID]]
```

ファイル末尾はLF一つで終了する。

IGCは次を追加または変更しない。

* `synapse_id`
* `synapse_facets`
* `integration_state`
* `integration_target_id`
* `integration_source_ids`
* `igc_sources`
* UUID
* 融合ID、融合グループID、所有者
* IGC統合・分離作業台
* 融合・分離・取消・復元・判断履歴の欄

## 6. 出力物理ファイル名

### 6.1 基底名

人間向け表記から、物理ファイル名の基底名を次の順序で作る。

1. Tagsだけ、先頭の`#`または`＃`を一文字除く。MentionsとLocationsは先頭記号を除かない。
2. `\ / : * ? " < > |`を一文字ごとに`_`へ置換する。
3. CRとLFを空白へ置換する。
4. 前後の空白を除く。
5. 先頭から100文字に制限する。
6. 空文字になった場合は`unnamed`とする。

この処理は物理保存名を作るためだけに行い、人間向け表記と統合キーは変更しない。

### 6.2 macOS上の物理名衝突

カテゴリごとに、基底名をUnicode NFDへ変換して`casefold()`した値を、macOS上の物理名衝突判定キーとする。

異なる人間向け表記が同じ衝突判定キーを持つ場合、そのグループに属する全カードの基底名へ次の接尾辞を付ける。

```text
--{12桁の小文字16進数}
```

12桁は次のUTF-8文字列のSHA-256から先頭12桁を取得する。

```text
{カテゴリ性質値}\0{人間向け表記}
```

カテゴリ性質値は次に固定する。

| カテゴリ | カテゴリ性質値 |
|---|---|
| Tags | `Synapse/Tag` |
| Mentions | `Synapse/Mention` |
| Locations | `Synapse/Location` |

例えば`#ABC`と`#abc`は統合せず、別々の決定的な物理名で同じTagsディレクトリへ保存する。

接尾辞付与後の物理名もNFD＋`casefold()`で一意でなければ異常終了する。接尾辞は物理ファイル衝突を避けるための保存名であり、Synapse IDまたは意味融合IDとしてカード本文へ書き込まない。

### 6.3 出力ファイル

出力ファイルは次の場所へ保存する。

```text
output_IGC/Instagram_Logs/Synapses/{カテゴリディレクトリ}/{物理名}.md
```

人間向け表記、Frontmatterの`aliases`、H1、カテゴリ固有値は、安全名に置き換えない。

## 7. SystemLogs出力

### 7.1 項目集合

IGCの一覧には、対応カテゴリの出力Synapseカードを一項目ずつ載せる。

一覧の人間向け表記集合とカードの人間向け表記集合は完全一致しなければならない。入力3一覧を単純連結せず、5.1の統合キーごとに一項目を出力する。

### 7.2 出力形式

カテゴリごとの見出しを次に固定する。

| カテゴリ | 見出し |
|---|---|
| Tags | `# ハッシュタグ一覧 - IGC統合` |
| Mentions | `# メンション一覧 - IGC統合` |
| Locations | `# 場所一覧 - IGC統合` |

一覧本文は次の形式とする。

```markdown
# 一覧名 - IGC統合

初期状態で全て採用（`- [x]`）です。除外したい項目は `- [x]` を `- [ ]` にしてください。

---

- [x] [[Instagram_Logs/Synapses/カテゴリ/物理名|人間向け表記]]
出現回数: N回

出現投稿:
  - [[投稿ID]]
---
```

出力時のチェック状態は全項目`- [x]`とする。

`N`は、統合後カードの関連投稿リンク数とする。一覧の出現投稿は、統合後カードと同じ値・同じ順番で出力する。

### 7.3 項目順

一覧項目は、次の順番で安定して出力する。

1. 出現回数の降順
2. 同数の場合は、人間向け表記をPython文字列として比較した昇順

この順番は意味、優先度、融合候補を表さない。

## 8. 入力不一致と異常終了

### 8.1 全処理を停止する条件

少なくとも次の場合は全カテゴリの恒久出力を変更せず、終了コード1で終了する。

* 必須入力ディレクトリまたは一覧ファイルがない。
* UTF-8で読み取れない。
* カードの必須構造、必須値、カテゴリが不正である。
* 同一系統・同一カテゴリに同じ人間向け表記のカードが複数ある。
* 入力一覧に`- [ ]`の項目がある。
* 共通情報が一致しない。
* カードと一覧の項目、物理リンク先、出現回数、関連投稿が一致しない。
* カード内、一覧内、または同一統合キーの系統間に完全同一投稿リンクが重複する。
* 出力物理名が一意にならない。
* 準備出力の検証に失敗する。
* 置換または復旧に失敗する。

一カテゴリで異常を見つけた後に、別カテゴリだけを恒久反映しない。

### 8.2 禁止する自動修復

異常時に次を行わない。

* どれか一系統の共通情報を正解として選ぶ。
* 空欄、人間の追記、カテゴリ固有情報を上書きする。
* 一覧だけの項目からカードを生成する。
* カードだけの項目を一覧から黙って削る。
* 関連投稿を推測で追加、削除、重複除去する。
* 表記またはカテゴリを変更して統合を続行する。

### 8.3 報告内容と再開条件

異常時は、可能な範囲で次をJSON結果に含める。

* 処理段階
* カテゴリ
* 入力系統
* 人間向け表記
* 物理ファイルまたは一覧ファイルのパス
* 対象投稿リンク
* 不一致の種類
* 人間が確認できる説明

入力または上流成果物の異常は、対象のIGP・IGR・IGS成果物を確認し、カードと一覧が一致した後にIGCを最初から再実行する。既存`output_IGC`を編集して入力不一致を吸収しない。

## 9. 検証仕様

### 9.1 入力検証

カテゴリ・入力系統ごとに次を確認する。

* カード件数と一覧項目件数が一致する。
* 人間向け表記集合が一致する。
* カードと一覧の物理リンク先が対応する。
* カードと一覧の関連投稿列が一致する。
* 出現回数が関連投稿数と一致する。
* 重複カード、重複一覧項目、重複投稿リンクがない。

### 9.2 Synapse出力検証

カテゴリごとに次を確認する。

* 出力の人間向け表記集合が、3入力の完全一致和集合と一致する。
* 出力件数が完全一致和集合件数と一致する。
* 各共通情報が、対応する入力の共通情報と完全一致する。
* 各関連投稿列が、IGP、IGR、IGSの順に加えた入力リンク列と一致する。
* 一系統だけのカードも存在する。
* 表記違いとカテゴリ違いが別カードのまま存在する。
* 出力物理名がmacOSのNFD＋`casefold()`で一意である。
* UUID、融合状態、融合用作業台が追加されていない。

### 9.3 SystemLogs出力検証

カテゴリごとに次を確認する。

* 一覧項目件数と出力カード件数が一致する。
* 一覧とカードの人間向け表記集合が一致する。
* 各一覧リンクが対応するカテゴリと物理ファイルを指す。
* 出現回数、出現投稿の値と順番が対応カードと一致する。
* 一覧項目が7.3の順番である。

### 9.4 対象外保護

恒久出力6か所以外は変更しない。ただし、本仕様で定めたリポジトリ内の一時準備領域と一時バックアップ領域への書き込みは許可する。

実行前後に、少なくともIGP・IGR・IGSの入力9ディレクトリと9一覧、および恒久出力6か所以外の`output_IGC`既存パスが変更されていないことを確認する。

## 10. 安全な出力更新

### 10.1 準備領域

最終出力へ直接書き込まず、リポジトリ直下に作成した一時ディレクトリ内へ6出力を生成する。

一時ディレクトリは、OSが重複しない名前を割り当てる。固定名の既存フォルダを作業領域として再利用しない。

準備領域で9章の全検証が成功するまで、既存`output_IGC`を変更しない。

### 10.2 ドライラン

`--dry-run`指定時も準備領域で生成と検証を行うが、恒久出力を変更しない。終了時に一時準備領域を削除する。

### 10.3 置換と復旧

検証成功後、既存の対象6か所だけを一時バックアップ領域へ移し、準備済み6出力と置き換える。

置換途中に失敗した場合は、その実行で設置した新出力を除去し、バックアップから対象6か所をすべて復元する。復元できなかった場合は成功として終了せず、残ったパスと状態を報告する。

置換成功後に一時バックアップを削除する。一時領域を削除できない場合は、恒久出力の検証結果と一時領域の残存を分けて報告する。

### 10.4 実行引数

メイン実行コードは次の引数を受け取る。

| 引数 | 必須 | 内容 |
|---|---|---|
| `--repo-root PATH` | 任意 | リポジトリルート。省略時はメイン実行コードの配置からリポジトリルートを決定する。 |
| `--dry-run` | 任意 | 準備出力の生成と検証だけを行い、恒久出力を置換しない。 |

実行時のカレントディレクトリに依存して入力・出力パスを決めない。

## 11. 実行結果

### 11.1 正常時

正常時は終了コード0とし、標準出力へ一つのJSONオブジェクトを出力する。

JSONは少なくとも次を持つ。

| 項目 | 型 | 内容 |
|---|---|---|
| `mode` | string | `dry-run`または`replace` |
| `ok` | boolean | 全検証の成否 |
| `replaced` | boolean | 恒久出力を置換したか |
| `inputs` | object | 3入力系統のパス |
| `categories` | object | カテゴリ別結果 |
| `categories.*.source_card_counts` | object | 系統別カード件数 |
| `categories.*.source_list_counts` | object | 系統別一覧件数 |
| `categories.*.input_union_count` | integer | 完全一致和集合件数 |
| `categories.*.multi_source_item_count` | integer | 複数系統に存在した統合キー数 |
| `categories.*.output_count` | integer | 出力カード件数 |
| `categories.*.related_post_count` | integer | 出力関連投稿リンク総数 |
| `categories.*.validation` | object | 検証項目ごとのboolean |

### 11.2 異常時

異常時は終了コード1とし、標準出力へ一つのJSONオブジェクトを出力する。

正常時の項目に加えて`error`オブジェクトを置き、少なくとも`type`、`phase`、`message`を持たせる。特定できる場合は`category`、`source`、`display_name`、`path`、`post_link`を含める。

人間向け表記または投稿リンクを報告する場合も、エラー箇所の確認に必要な対象だけに限定し、一覧全体を標準出力へ複製しない。

# IGC統合仕様書 v1.0 — 役割・入力形式

## 1. 目的

本仕様書は、[[02_IGC統合設計書v1.0]]と、分割後の正式設計書で確定したIGCの責任に基づき、IGP・IGR・IGSのSynapseカードとSystemLogs一覧を`output_IGC`へ統合する具体的な規則を定義する。

IGCが行うのは、同一カテゴリ・完全同一表記のカードを一件へ整理し、関連投稿を集め、カードと一覧を対応させる処理だけである。表記が異なるカードの意味融合、カテゴリ横断融合、大きなメモ、共通作業台、融合・分離・判断履歴は扱わない。

## 2. 実行範囲

### 2.1 入力系統

入力系統と処理順を次に固定する。

1. `output_IGP`
2. `output_IGR`
3. `output_IGS`

カテゴリごとの入力は次のとおりとする。

| カテゴリキー | Synapseカード | SystemLogs一覧 |
|---|---|---|
| `tags` | 各系統の`Instagram_Logs/Synapses/Tags/*.md` | 各系統の`Instagram_Logs/SystemLogs/ハッシュタグ一覧.md` |
| `mentions` | 各系統の`Instagram_Logs/Synapses/Mentions/*.md` | 各系統の`Instagram_Logs/SystemLogs/メンション一覧.md` |
| `locations` | 各系統の`Instagram_Logs/Synapses/Locations/*.md` | 各系統の`Instagram_Logs/SystemLogs/場所一覧.md` |

入力ディレクトリと一覧ファイルは、カードまたは一覧項目が0件の場合も存在しなければならない。0件のLocations等は正常な空入力として扱う。

macOSがファイル名をUnicode正規化して保存する場合、固定された一覧ファイル名はNFCまたはNFDの正規化差だけを同じ物理名として探索してよい。同じ正規化名に複数ファイルが対応する場合は異常終了する。この扱いを、カードの人間向け表記の完全一致判定には適用しない。

### 2.2 恒久出力

恒久的に作成または置換する対象は次の6か所だけとする。

* `output_IGC/Instagram_Logs/Synapses/Tags`
* `output_IGC/Instagram_Logs/Synapses/Mentions`
* `output_IGC/Instagram_Logs/Synapses/Locations`
* `output_IGC/Instagram_Logs/SystemLogs/ハッシュタグ一覧.md`
* `output_IGC/Instagram_Logs/SystemLogs/メンション一覧.md`
* `output_IGC/Instagram_Logs/SystemLogs/場所一覧.md`

### 2.3 対象外

次は入力、生成、変更、削除、補完の対象外とする。

* `output_IGX`
* Posts
* Reels
* Stories
* media
* Timeline
* RawData
* 実際のObsidian Vault
* iCloud配下

既存の`output_IGC`は、入力、正解、補完元、差分吸収元として使用しない。

## 3. Synapseカードの入力形式

### 3.1 文字コードと改行

カードと一覧はUTF-8として読み取る。デコードできない場合は異常終了する。

改行コードは読み取り時にLFへ統一して比較する。Unicode文字、空白、インデント、コメント、大文字小文字、全角半角は正規化しない。

### 3.2 完全一致に使用する人間向け表記

物理ファイル名は統合キーに使用しない。カード内の次の値を、人間向け表記として読み取る。

| カテゴリ | 必須の値 | 例 |
|---|---|---|
| Tags | `hashtag_note.hashtag` | `#SampleTagA` |
| Mentions | `mention_note.mention` | `@sample_account_001` |
| Locations | `location_note.location` | `大阪城` |

値は引用符を外した文字列として取得する。前後空白の削除、Unicode正規化、大文字小文字変換、記号変換、別名解決は行わない。

同じカードについて、次の四つが完全に同じ文字列でなければならない。

1. Frontmatterの`aliases`にある唯一の値
2. 本文先頭のH1見出し
3. 上表のカテゴリ固有値
4. 対応するSystemLogs項目のWikiリンク表示名

Frontmatterの`tags`は、カテゴリごとに`Synapse/Tag`、`Synapse/Mention`、`Synapse/Location`のいずれか一つと一致しなければならない。

必須値がない、複数ある、値の型が文字列ではない、四つの値が一致しない、カテゴリが一致しない場合は異常終了する。

### 3.3 カードの共通情報と関連投稿

カードには`## 関連投稿`見出しを一つだけ置く。この見出しより前を「共通情報」、見出しより後を「関連投稿領域」とする。

関連投稿領域は、空行と次の形式のWikiリンクだけを許可する。

```markdown
[[投稿ID]]
```

別名、見出し、ブロック参照、埋め込み、Markdown箇条書き、通常文章を関連投稿領域に置かない。投稿リンクが0件のカードは異常とする。

同じカード内に同一投稿リンクが複数ある場合は異常終了する。

### 3.4 同一入力系統内の一意性

一つの入力系統・一つのカテゴリ内で、同じ人間向け表記を持つカードは一枚だけでなければならない。

物理ファイル名が異なっていても、人間向け表記が同じカードが複数存在する場合は、どちらかを選ばず異常終了する。

## 4. SystemLogs一覧の入力形式

### 4.1 一覧項目

一覧項目は次の行で開始する。

```markdown
- [x] [[物理リンク先|人間向け表記]]
```

`- [X]`も採用状態として読み取る。入力側に`- [ ]`が一件でもある場合は、人間による除外判断が存在する可能性があるため、その項目を無視または自動除外せず、対象の入力系統、カテゴリ、人間向け表記を報告して全処理を停止する。

入力一覧の全項目が`- [x]`または`- [X]`である場合だけ、Synapseカードと一覧に存在する全項目を照合・統合する。

人間向け表記に`.md`を付けない。物理リンク先は、対応する入力Synapseカードの`.md`を除いた物理ファイル名と一致しなければならない。フォルダ付きリンクの場合は最後のパス要素を比較する。

同じ一覧内に同じ人間向け表記が複数ある場合は異常終了する。

### 4.2 出現回数と出現投稿

各項目は次の形式を持つ。

```markdown
出現回数: N回

出現投稿:
  - [[投稿ID]]
```

`N`は0以上の整数として解析し、出現投稿リンク数と一致しなければならない。同じ項目内に同一投稿リンクが複数ある場合は異常終了する。

出現投稿リンクが0件の項目は異常とする。

### 4.3 カードと一覧の入力照合

統合前に、入力系統・カテゴリごとに次をすべて確認する。

* カードの人間向け表記集合と一覧の人間向け表記集合が一致する。
* 一覧の物理リンク先が、同じ人間向け表記を持つ実カードを指す。
* カードの関連投稿リンク列と一覧の出現投稿リンク列が、件数、値、順序とも一致する。
* 一覧の出現回数が、カードの関連投稿リンク数と一致する。

一覧だけ、カードだけ、リンク先不一致、関連投稿不一致、件数不一致を自動修復しない。いずれか一つでも不一致があれば、統合出力を作らず異常終了する。

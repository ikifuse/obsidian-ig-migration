# 01_IGC統合企画書v1.0

## この企画で実現したいこと

`output_IGP`、`output_IGR`、`output_IGS` に個別出力されたハッシュタグ、メンション、場所の Synapse を、Obsidian へ投入する前に整理したい。

3系統に同じものが存在する場合、単純合計にはせず、重複を1件として `output_IGC` にまとめたい。

1系統だけに存在するものは欠落させず、人生記録と投稿リンクを支える重要な情報として残したい。

## 出力したいもの

今回の IGC 統合で新しく作成・更新する対象は、次の6か所だけとする。

* `output_IGC/Instagram_Logs/Synapses/Tags`
* `output_IGC/Instagram_Logs/Synapses/Mentions`
* `output_IGC/Instagram_Logs/Synapses/Locations`
* `output_IGC/Instagram_Logs/SystemLogs/ハッシュタグ一覧.md`
* `output_IGC/Instagram_Logs/SystemLogs/メンション一覧.md`
* `output_IGC/Instagram_Logs/SystemLogs/場所一覧.md`

Posts、Reels、Stories、media、timeline、RawData は、今回の統合処理では作成も変更もしない。

`output_IGX` は今回の対象外とする。

既存の `output_IGC` は正解、入力、補完元として扱わない。

## 一覧で実現したいこと

ハッシュタグ一覧、メンション一覧、場所一覧も、IGP・IGR・IGS の3系統をまとめたい。

同じ項目が複数系統にある場合、その項目につながる投稿IDをすべて集めたい。

同じ投稿IDが重なった場合は1件として扱い、出現回数も重複を除いた投稿ID件数にしたい。

投稿IDそのものや投稿メモのファイル名は変更したくない。

## Synapseを作る本当の目的

Synapse は、単なるタグ・場所・アカウントの一覧ではない。

Instagram に残された事実と、後から人間が追記する意味、記憶、位置、連絡先、関係、判断を集め、将来AIも理解できる第二の脳へ育てるためのメモである。

統合によって情報を減らすのではなく、1つの Synapse メモが複数の専門情報を吸収し、より多くの情報を持つメモへ成長してほしい。

例えば、同じ「天満天神繁昌亭」がタグと場所に存在する場合、統合後の1メモにはタグとしての意味と、場所としての住所・緯度経度・チェックイン情報の両方を持たせたい。

メンションと場所、タグとメンションなど、将来別の組み合わせが同じ存在だと人間が判断した場合にも対応できるようにしたい。

ただし、今回の IGC 統合でそこまで進めるわけではない。今回は、3系統から渡された初期Synapseメモを壊さずに整理し、将来人間が統合・分離を判断できる状態を整えるところまでを目的とする。

## 3種類の統合

現時点では、統合には少なくとも次の3種類があると考える。

* IGP・IGR・IGS に存在する完全に同じ Synapse を、Obsidian投入前に1件にする統合
* 名前は違うが同じ意味だと人間が判断したものを、将来UIで1つにする統合
* Tags・Mentions・Locations のカテゴリをまたぐ同じ存在を、将来UIで1つにする統合

最初の完全重複は IGC 工程で整理したい。

意味判断を伴う統合は自動化せず、将来 Memory Synapse DB の UI で人間が判断した後に実行したい。

## 将来統合のために今回用意したいもの

今回 `output_IGC` に出す Synapse には、将来の深い統合と分離に備えた受け皿や作業台を最初から設置したい。

将来統合した時には、Tags・Mentions・Locations が持つ専門情報を失わず、1つの成長型 Synapse メモへ集めたい。

統合前の情報、どのメモを統合したか、なぜ統合したか、どの投稿リンクを変更したかを後から確認できるようにしたい。

間違って統合した場合は、元の Synapse メモ、専門情報、投稿リンクを分離・復元できるようにしたい。

## 人間とUIの役割

システムやAIは、名前が似ている、場所が近い、同じ言葉を含むという理由だけで意味統合しない。

統合・分離の最終判断は人間が行う。

将来のUIは、統合前の変更内容を表示し、人間が確認してから実行できるものにしたい。

Synapse の実体と判断履歴は、プラグイン内部だけに閉じ込めず、人間が読める Markdown として残したい。

## 安全に関する希望

1投稿1メモの構造を維持したい。

投稿IDは変更しない。

元の Instagram 事実を失わない。

統合処理を実際の Obsidian Vault や iCloud 上で直接行わない。

既存の IGP・IGR・IGS の成果物と実行コードを壊さない。

今回の統合実行と、将来UIで行う意味統合を混同しない。

## 現時点の位置付け

この企画書に書かれた内容は、IGC統合の新しい出発点である。

実現方法、メモ構造、識別方法、統合履歴、分離方法、検証方法は、後続の設計書で問題点と成立条件を確認する。

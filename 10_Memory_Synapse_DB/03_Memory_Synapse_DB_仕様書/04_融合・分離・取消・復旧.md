# Memory Synapse DB 仕様書 v1.0 — 融合・分離・取消・復旧

## 第7章 融合操作

### 7.1 操作開始

ブラウザー確認画面とObsidianプラグインの両方で、次の入口を用意する。

- カードを別カードへドラッグ＆ドロップする
- カード内の`融合へ追加`から対象カードを選ぶ

ドラッグした側と受け入れる側のどちらが大きなカードになるかを、操作だけで確定しない。

### 7.2 構成員の展開

融合対象が単独カードなら、その一枚を候補集合へ加える。大きなカードまたはその構成員なら、所属する融合単位の全カードを候補集合へ加える。

候補集合は重複を除去して平らな一つの配列にする。融合前の受け皿を入れ子で残さない。

### 7.3 大きなカードの推奨

種類ごとの優先順位は次のとおりとする。

1. Mention
2. Location
3. Tag

候補集合内で最も優先順位が高い種類を推奨対象とする。同じ最高順位のカードが複数ある場合は、受け入れる側のカードまたは受け入れる側グループの現在の大きなカードを第一候補とする。

第一候補にできない場合は、同順位のカードを全て「推奨」として表示し、自動選択しない。

### 7.4 確認ポップアップ

融合のたびに、`大きなカードはどれにしますか？`と表示する。ポップアップには次を表示する。

- 全候補カードの種類とカード名
- 現在の大きなカード
- 推奨候補と推奨理由
- 融合後の全構成員
- 変更されるMarkdownファイル
- `融合する`と`キャンセル`

候補の一枚を人間が選ぶまで`融合する`を有効にしない。推奨候補は変更できる。

### 7.5 書き込み

確認後、次の順で更新案を作る。

1. 選択した大きなカードを候補集合の先頭とする。
2. 残りを、既存の表示順を維持して`members`へ入れる。
3. 選択した大きなカードへ`## Memory Synapse`を追加または更新する。
4. 融合前に大きなカードだった別カードから`## Memory Synapse`だけを除く。
5. 全カードの`## 手書き情報`とカテゴリ固有情報を維持する。

変更対象を全て検証してから書き込む。一件でも読込不能、リンク切れ、解析不能、多重所属がある場合は、どのファイルも変更しない。

## 第8章 大きなカードの変更

### 8.1 通常の変更

大きなカード内の`大きなカードを変更`を押すと、現在の全構成員を候補とする第7.4節と同じ選択ポップアップを表示する。

確認後、新しい大きなカードへ`## Memory Synapse`を移し、旧大きなカードから同見出しだけを除く。全構成員と各カードの手書き情報は維持する。

### 8.2 表示モード

新しい大きなカードに手書き情報がある場合は、確認ポップアップで次のどちらを主表示にするか選ぶ。

- 元の個別カード情報
- 手書き情報

手書き情報がない場合は`source`固定とする。旧大きなカードの手書き情報は削除しない。

## 第9章 分離操作

### 9.1 通常構成員の分離

受け皿内の各構成員に`このカードを分離`を表示する。大きなカード以外を選んだ場合は、確認後、そのWikiリンクだけを`members`から除く。

分離したカードのファイル内容は変更しない。残る全構成員数が二枚以上なら、現在の大きなカードを維持する。

### 9.2 大きなカード自身の分離

受け皿内の大きなカード自身にも`このカードを分離`を表示する。選択した場合は、残る構成員を表示し、`残す大きなカードはどれにしますか？`と確認する。

残る構成員が二枚以上の場合は、人間が次の大きなカードを選択する。推奨規則は第7.3節と同じとする。

確認後、旧大きなカードから`## Memory Synapse`だけを除き、新しい大きなカードへ残る構成員を保存する。旧大きなカードのカテゴリ固有情報と手書き情報は維持する。

### 9.3 一枚だけ残る場合

分離後に一枚だけ残る場合は、その一枚を単独カードへ戻す。`## Memory Synapse`は残さず、新しい大きなカードの選択も求めない。

### 9.4 融合全体の解体

大きなカード内の`融合をすべて解体`を押すと、全構成員と影響を表示する。確認後、大きなカードから`## Memory Synapse`だけを除く。

全個別カードのカテゴリ固有情報、関連投稿、手書き情報は変更しない。

## 第10章 取消・失敗・復旧

### 10.1 操作単位

次をそれぞれ一回の操作として扱う。

- 融合
- 大きなカードの変更
- 一枚の分離
- 融合全体の解体
- 手書き情報の保存
- 表示モードの変更

### 10.2 操作前検証

書き込み前に次を確認する。

- 全対象ファイルが存在する
- 全対象ファイルを読める
- 管理対象YAMLを解析できる
- 対象リンクを一意に解決できる
- 多重所属と入れ子がない
- 読み取り後に対象ファイルが別操作で変更されていない

一項目でも失敗した場合は書き込まず、原因と対象ファイルを表示する。

### 10.3 書き込みとロールバック

書き込み前に、変更対象ファイルごとに次をローカル操作履歴へ保持する。

- Vault相対パス
- 操作前の管理対象見出し内容
- 操作後の管理対象見出し内容
- 操作日時
- 操作種類
- 操作前後の内容照合値

複数ファイルの書き込み途中で失敗した場合は、書き込み済みファイルの管理対象見出しを操作前へ戻す。復旧にも失敗した場合は成功表示を出さず、変更済み・未変更・復旧失敗のファイルを分けて表示する。

### 10.4 直前操作の取消

画面に`元に戻す`を用意する。取消対象は最後に成功したMemory Synapse DB操作とする。

現在の管理対象見出しが履歴の操作後内容と一致する場合だけ取消を実行する。操作後に人間または別処理が編集している場合は自動で上書きせず、差分を表示して中止する。

操作履歴は`<Vault>/.obsidian/plugins/memory-synapse-db/operation-history.json`へ保存し、成功した直近100操作を保持する。101件目を保存した後に最古の一件を除く。履歴は知識関係の正本にはせず、Git管理、外部送信、SystemLogsへの混入の対象にしない。

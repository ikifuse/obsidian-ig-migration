# Memory Synapse DB 仕様書 v1.0 — 役割・対象データ

## 第1章 本仕様書の役割

### 1.1 目的

本仕様書は、`../02_Memory_Synapse_DB_設計書/00_設計書目次.md`とそこに記載された設計4冊を、ブラウザー上の確認画面とObsidianプラグインへ実装できる具体的な規則へ変換する。

本仕様書が扱うのは、個別カードの表示、融合、追加、分離、大きなカードの選択と変更、手書き情報、取消、復旧である。IGP・IGR・IGS・IGCの抽出・変換仕様は変更しない。

### 1.2 優先順位

判断の優先順位は次のとおりとする。

1. `01_IG移行企画書v1.0.md`
2. `01_Memory_Synapse_DB_企画書v2.1.md`
3. `../02_Memory_Synapse_DB_設計書/00_設計書目次.md`とそこに記載された設計4冊
4. 本仕様書
5. コードおよび成果物

旧`04_IG移行仕様書/06_Memory_Synapse_DB_仕様書v1.2.md`のうち、個別カード形式と自動で意味を確定しない規則は維持する。「共通作業台は未確定」「融合・分離は将来工程」という記述は、本仕様書で置き換える。

### 1.3 実装順序

1. 本仕様に基づく確認画面をブラウザー上に作る。
2. オーナーが検証用カードで画面と操作を確認する。
3. 確認によって仕様変更が必要になった場合は、本仕様書を修正して再承認する。
4. 仕様承認後、ObsidianプラグインをTypeScriptで実装する。
5. Obsidian上のMarkdown実物とWikiリンクで最終確認する。

ブラウザー確認だけを完成としない。ブラウザー確認中は、実際のVaultおよび`output_IGC`を書き換えない。

## 第2章 対象データ

### 2.1 入力ルート

入力は、検証済みの次のディレクトリとする。

```text
output_IGC/Instagram_Logs/Synapses/
├── Tags/*.md
├── Mentions/*.md
└── Locations/*.md
```

実際のVaultへ導入した後は、Vault内の同じ相対構造を対象とする。対象ルートはプラグイン設定で一つだけ選択し、初期値を`Instagram_Logs/Synapses`とする。

IGX、SystemLogs、投稿、Reel、Story、RawData、mediaは、融合関係の書き込み対象にしない。

### 2.2 個別カードの種類

カード種類はVault相対パスで判定する。

| パス | 種類 | 表示名 |
|---|---|---|
| `*/Synapses/Tags/*.md` | `tag` | Tag |
| `*/Synapses/Mentions/*.md` | `mention` | Mention |
| `*/Synapses/Locations/*.md` | `location` | Location |

対象外のパスは自動判定しない。将来種類を増やす場合は仕様変更として扱う。

### 2.3 個別カードの保持条件

- 個別カードは融合後も同じMarkdownファイルとして残す。
- ファイル名、配置フォルダー、Frontmatter、カテゴリ固有情報、関連投稿を、融合だけを理由に変更しない。
- 融合専用カード、受け皿専用カード、手書き専用カードを新規作成しない。
- 出現回数、投稿回数を個別カードへ追加しない。
- 投稿Frontmatterへ、カード側の位置情報、住所、融合状態、手書き情報を同期しない。

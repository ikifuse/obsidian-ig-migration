# Memory Synapse DB 仕様書 v1.0 — 個別カード・融合状態

## 第3章 既存の個別カード形式

### 3.1 共通構成

既存カードのFrontmatter、見出し、情報YAML、`## 関連投稿`を読み取る。既存の未知のFrontmatter項目、本文、コメント、空欄を削除しない。

Memory Synapse DBが追加・更新できる本文見出しは、次の二つだけとする。

- `## Memory Synapse`
- `## 手書き情報`

追加位置は`## 関連投稿`とそのWikiリンク群の後ろとする。両方を持つ場合は`## Memory Synapse`、`## 手書き情報`の順に置く。既に対象見出しがある場合は、新しい同名見出しを追加せず、その見出し内の管理対象YAMLだけを更新する。

### 3.2 Tagカード

Tagカードのカテゴリ固有情報は次の形式を維持する。

````markdown
```yaml
hashtag_note:
  hashtag: "#SampleTagA"
  note: null
```

## 関連投稿

[[2024-01-06-12-00-00_IG_0006]]
````

`hashtag`は抽出された元表記を変更しない。`note`は人間が育てる既存のカテゴリ固有欄として保持する。

### 3.3 Mentionカード

Mentionカードのカテゴリ固有情報は次の形式を維持する。

````markdown
```yaml
mention_note:
  mention: "@sample_account_001"
  name: null
  phone: []
  web:
    - "https://www.instagram.com/sample_account_001/"
  note: null
```

## 関連投稿

[[2022-05-11-17-20-55_IG_836]]
````

`mention`は抽出された元の`@文字列`を変更しない。生成済みInstagram URLは、現在の所有者や本人性を保証する値として扱わない。

### 3.4 Locationカード

Locationカードのカテゴリ固有情報は次の形式を維持する。

````markdown
```yaml
location_note:
  location: "サンプル地点A"

geo:
  lat: 35.000001
  lng: 135.000001
  alt: null

address:
  full: null
  components:
    country: null
    prefecture: null
    city: null
    district: null
    street: null
    postal_code: null

activity_id: "activity_000001"

source_files:
  - "[[activity_000001.gpx]]"

note: null
```

## 関連投稿

[[2022-09-18-14-53-08_IG_641]]
````

移行時点の`location_note`、`geo`、`address`、`activity_id`、`source_files`、`note`を、融合操作で変更しない。人間が後から補正する値は、第6章の手書き情報へ保存する。

## 第4章 融合状態の保存形式

### 4.1 保存場所

融合状態は、大きなカードとして選ばれた既存カードだけに置く`## Memory Synapse`で管理する。大きなカード以外の構成員には同見出しを置かない。

大きなカード自身は、そのファイルに`## Memory Synapse`が存在することで識別する。自分自身へのWikiリンクは保存しない。

### 4.2 YAML形式

````markdown
## Memory Synapse

```yaml
memory_synapse:
  schema_version: 1
  display_mode: source
  members:
    - "[[Instagram_Logs/Synapses/Locations/サンプル地点B|サンプル地点B]]"
    - "[[Instagram_Logs/Synapses/Tags/#サンプルタグB|#サンプルタグB]]"
```
````

| 項目 | 型 | 必須 | 規則 |
|---|---|---|---|
| `schema_version` | 整数 | 必須 | v1.0では`1`固定 |
| `display_mode` | 文字列 | 必須 | `source`または`handwritten` |
| `members` | Wikiリンク文字列の配列 | 必須 | 大きなカード以外の全構成員。二枚融合なら一件 |

`members`に同じ解決先を重複登録しない。空配列の`memory_synapse`は保存しない。

### 4.3 Wikiリンク

- WikiリンクはObsidianのリンク生成機能を使い、現在のカードから対象カードへ解決できる形で生成する。
- 同名ファイルが存在する場合はVault相対パスを含める。
- 表示名には対象カードの現在のファイル名から`.md`を除いた値を使う。
- リンク解決先が存在しない場合は融合を実行しない。
- Obsidianによる改名・移動でリンクが更新された場合、更新後のリンクを正とする。
- 一般の本文リンクや`## 関連投稿`のリンクを、融合構成員として解釈しない。

### 4.4 大きなカードの判定

有効な`memory_synapse`を持つファイルを大きなカードとする。大きなカードと、その`members`を合わせた集合を一つの融合単位とする。

大きなカードを含む全構成員数は`members`件数に一を加えた値とする。融合グループID、大きなカード専用IDは追加しない。

### 4.5 多重所属の禁止

プラグインは全ての`memory_synapse.members`を読み、カードから所属する大きなカードへの索引をメモリ上に作る。

次の状態を検出した場合は書き込まず、対象ファイルを表示して修復を求める。

- 同じカードが複数の大きなカードの`members`に存在する
- 大きなカードが別の大きなカードの`members`に残った入れ子状態
- `members`に大きなカード自身が含まれる
- リンク切れ、YAML解析失敗、未対応`schema_version`

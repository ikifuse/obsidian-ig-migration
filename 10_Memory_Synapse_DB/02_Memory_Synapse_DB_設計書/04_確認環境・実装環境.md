# Memory Synapse DB 設計書 v1.0 — 確認環境・実装環境

## 第7章 確認環境と実装環境

### 7.1 ブラウザー上の設計確認

画面と操作は、Obsidianプラグインへ固定する前にブラウザー上の検証環境で確認する。安全な検証用データまたは複製を用い、日常利用データを直接変更しない。

少なくとも、同じカテゴリ同士、異なるカテゴリ同士、大きなカードへの追加、大きなカード同士、通常表示と手書き表示、大きなカード自身の分離を確認する。

ブラウザーで使いやすく見えることだけを完成根拠にしない。確認結果を設計と仕様へ反映した後、Obsidianプラグイン、Wikiリンク、Markdown実物で再確認する。

### 7.2 Obsidianプラグイン

Obsidianプラグインは、人間が`output_IGC`の個別カードと出どころを確認し、Markdownを実体として知識を育てる最終操作入口とする。

プラグインは、融合、追加、分離、大きなカードの選択と変更、手書き情報の追記と編集、操作前の影響確認、取消と復旧を扱う。抽出工程で欠落した事実を推測して補わず、似ているという理由だけで意味を自動確定しない。

ブラウザー上で画面と操作を確認し、仕様を承認した後、ObsidianプラグインをTypeScriptで実装する。この判断はプラグインの実装言語だけを固定し、画面、保存形式、処理手順、ビルド方法等の具体化を先取りしない。

### 7.3 ローカルと外部AI

知識の正本は、オーナーが管理するiMac内のMarkdownと関係情報とする。プラグインや外部サービスは、オーナーの確認なしに個人データを外部へ送信しない。

外部AIの利用自体は禁止しない。利用する場合は、対象、目的、送信する情報をオーナーが確認して判断する。AIは候補提示や比較を支援できるが、意味関係を自動確定しない。

## 第8章 仕様へ渡す内容

仕様書では、少なくとも次を実装者が推測しなくてよい粒度で定める。

- 個別カードと融合関係を識別する具体的な項目と形式
- Wikiリンクの配置、更新、改名・移動時の扱い
- 大きなカードと全構成員を表示する具体的な方法
- 通常表示、短縮表示、手書き表示の画面と切替え
- 融合、追加、分離、大きなカード変更、全体解体の操作手順
- 候補提示、選択、実行前確認、完了表示、エラー表示
- 手書き情報と補正情報の保存、編集、引継ぎ
- 途中失敗、取消、復旧、再実行の単位と合否条件
- ブラウザー検証とObsidian実物検証で使うケースと判定条件

現在の設計では、具体的な保存項目、画面配置、操作部品、処理アルゴリズムは確定しない。ブラウザー上の確認で設計上の意味を変える必要が生じた場合は、仕様へ進めず、本設計書へ戻ってオーナーの承認を得る。

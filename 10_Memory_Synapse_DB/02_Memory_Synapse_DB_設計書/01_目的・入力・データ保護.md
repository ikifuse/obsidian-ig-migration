# Memory Synapse DB 設計書 v1.0 — 目的・入力・データ保護

## 第1章 本設計書の役割

### 1.1 目的

本設計書は、`01_Memory_Synapse_DB_企画書v2.1.md`で承認された内容を、Memory Synapse DB工程内部の構造、責任、データ間の関係、守る条件へ分解する。

プロジェクト全体の工程間の境界と共通原則は`02_IG移行設計書/00_設計書目次.md`と設計6冊が定める。本設計書は、その境界を変えず、Memory Synapse DB内部の設計正本として扱う。

具体的な項目名、データ型、保存位置、画面配置、ボタン、確認文言、処理順、判定式は仕様書で定める。

### 1.2 現在の到達点

検証済みの`output_IGC`をObsidianへ導入し、オーナーが次の動作を実物で確認できる状態を現在の到達点とする。

- 個別カードと関連投稿の確認
- 人間判断による融合と分離
- 大きなカードの選択と変更
- 大きなカードへの手書き情報の追記と編集
- 融合後の全個別情報の確認
- 誤操作や途中失敗からの復旧

ブラウザー上の確認は設計を具体化するための検証であり、Obsidian上の完成確認を置き換えない。

## 第2章 入力とデータ保護

### 2.1 入力

Memory Synapse DBは、IGP・IGR・IGSを入力とするIGCで生成・検証された`output_IGC`を受け取る。IGXは現在の入力と完走条件に含めない。

`output_IGC`はInstagram原本から作られた加工成果物であり、Instagram原本そのものではない。Memory Synapse DBは、上流工程の原本を変更しない。

### 2.2 個別カード

Tag、Mention、Location等の個別カードは、それぞれ独立したMarkdownファイルとして保持する。融合は、個別カードを一つのファイルへ物理的に結合したり、元ファイルを削除したりする処理ではない。

個別カードが移行時点で持つ種類、元表記、固有情報、関連投稿への関係を、融合、分離、表示の切替えによって失わせない。

後から人間が加えた意味、記憶、判断と、移行時点の情報を区別できる構造にする。後から確認・補正した位置情報等の具体的な保存先は、原本と加工成果物の区別を保つことを条件として仕様書で定める。

### 2.3 リンクを正本とする関係

融合関係は、MarkdownとObsidianのWikiリンクから、人間とAIが確認できる状態にする。プラグインだけが読める内部データを唯一の正本にしない。

一般的な関連リンクと融合関係は区別する。一つの個別カードを、同時に複数の大きなカードへ融合しない。複数の知識との一般的な関係は、融合とは別のリンクで表す。

## 第1章 システム共通仕様 (対応: [[02_システム全体構成・責任境界]] / [[04_データ取得・変換・出力設計]])

### 1.1 一方向パイプライン処理フロー
データ変換は一方向のパイプラインに限定する。処理順序は以下の通りとする。
1. **データ取込・正規化**: 原本JSON群の読み込み、および文字コード統一（UTF-8）と日時変換。
   * [差分反映: 実データには構造の異なる `posts.json` と `posts_1.json` が同一期間に存在し、ReelsとStoriesもトップレベルに特定のキーを持つ構造になっている事実を検出] (変更理由: 実データ監査結果に基づく事実)
   * **複数JSON重複時の共通原則**:
     * 複数の原本JSONが同一の投稿群を別スキーマで重複して保持している場合、単純な合算処理（単純追加による二重出力）を行ってはならない。
     * 実データ監査に基づき、情報量が最も豊富なJSONを「主データ」として定義し、これのみを移行処理の実行対象とする。その他の重複データは「重複・監査用データ」と位置づけ、実行対象外（監査・照合用）として保持する。
     * 重複データ（監査用データ）を移行処理で補完用途に使用する場合は、その必要性と補完ルールを個別仕様書で明示的に定義する。個別仕様に定義がない限り、重複データを自動マージ・自動補完しない。
     * 推測に基づく重複統合を実行コード側に実装してはならず、1つの物理的な投稿（イベント）から複数のMarkdownが二重に生成されることを確実に防止する。
2. **一時出力フォルダでの隔離生成**: 個別データ種別ごとの一時出力フォルダ（`output_IGP` / `output_IGR` / `output_IGS` 等）へMarkdownおよびメディアを出力。
3. **個別Synapseカードとタイムライン生成**: カテゴリ固有情報と育てる欄を持つ個別Synapseカード、およびタイムラインの出力。
4. **手動確認と工程確定**: 各工程の出力を隔離環境で検証し、問題がなければ同じ工程の正式出力（`output_IGP` / `output_IGR` / `output_IGS` / `output_IGX`）として確定する。
5. **IGC統合**: IGP・IGR・IGSの3工程が確定した後、専用IGC工程が3出力を入力として、`output_IGC` のSynapse 3ディレクトリとSystemLogs 3一覧だけを生成・置換する。IGP・IGR・IGSから `output_IGC` へ直接再実行してはならない。

## 第2章 データ構造・YAML仕様 (対応: [[03_データ構造・原本保持設計]] / [[04_データ取得・変換・出力設計]])

### 2.1 YAMLフロントマター・スキーマ
すべての移行先MarkdownノートのYAMLヘッダーは、以下の構造（4層YAML）を厳密に採用する。

| フィールド名 | データ型 | 必須 / 任意 | 初期値（デフォルト） | 説明 |
| :--- | :--- | :--- | :--- | :--- |
| **id** | string | 必須 | なし | タイムスタンプベースのユニークID（例: `"2026-06-24-07-50-12_IG_001"`） |
| **source** | string | 必須 | `"instagram"` | データソース識別子。 |
| **type** | string | 必須 | なし | データ種別（`Feed` / `Reels` / `Stories` / `Salvage` 等）。 |
| **content** | string / null | 任意 | `null` | メディア種別（動画の場合は `"video"`、画像の場合は `null` を指定）。 |
| **date** | string | 必須 | なし | 投稿のISO8601日付文字列（例: `"2026-06-24T07:50:12+09:00"`） |
| **created_at** | string | 必須 | なし | `date` と同一のISO8601文字列。 |
| **updated_at** | string | 必須 | なし | `date` と同一のISO8601文字列。 |
| **event_at** | string | 必須 | なし | `date` と同一のISO8601文字列。 |
| **duration** | integer / null | 任意 | `null` | 動画等の再生時間（秒）。不明時は `null`。 |
| **created_at_ms** | integer | 必須 | なし | 投稿日時のエポックミリ秒（`timestamp * 1000`）。 |
| **updated_at_ms** | integer / null | 任意 | `null` | 更新日時のエポックミリ秒。 |
| **event_at_ms** | integer / null | 任意 | `null` | イベント日時のエポックミリ秒。 |
| **tags** | array of string | 必須 | `[]` | 抽出されたハッシュタグのリスト。人間判断前の意味統合は行わない。 |
| **mentions** | array of string | 必須 | `[]` | 抽出されたメンション表記のリスト（接頭辞 `@` を付与）。実在アカウント確定は行わない。 |
| **links** | array of string | 必須 | `["[[instagram]]"]` | Wikiリンク形式の関係リンク。tagsやmentions等の関係ノードへのリンク。 |
| **group_id** | string / null | 任意 | `null` | グループ投稿等の識別ID。通常は `null`。 |
| **location** | dictionary | 必須 | 下記参照 | 位置情報オブジェクト。 |
| **instagram_id** | string | 必須 | なし | メディアファイル名等から抽出したInstagram固有の識別文字列。 |
| **original_filename**| string | 必須 | なし | 原本側の先頭メディアファイル名または原本JSONファイル名。 |
| **migration_version**| string | 必須 | `"v6.2"` | 本仕様の準拠バージョン。 |
| **created_from** | string | 必須 | `"Instagram export JSON"` | データの作成元説明。 |
| **media_count** | integer | 必須 | `0` | 投稿に含まれるメディアファイル数。 |
| **media** | array of dict | 必須 | `[]` | コピーされたメディア情報。各要素は `path` キーを持つ辞書。 |
| **emoji_original** | array of string | 必須 | `[]` | キャプション内から抽出された絵文字そのもののリスト。 |
| **emoji_normalized** | array of string | 必須 | `[]` | 絵文字の日本語名称表記（スペースはアンダースコアへ変換）。 |
| **raw_source_path** | string | 必須 | なし | 原本個別JSONへのWikiリンク（例: `"[[2026-06-24-07-50-12_IG_001.json]]"`）。 |
| **unparsed** | null | 必須 | `null` | 将来用の予備フィールド（`null`固定）。 |

時刻フィールドは、ストーリー等の自動分割や同一日時内の複数メディアを後から再整理できるよう、秒単位のISO8601文字列とミリ秒フィールドを併用する。

location:
  raw: string / null       # スポットの名前（未加工）
  normalized: string / null# 抽出後のスポットの名前
  geo:

```

#### 2.2.6 IGCの更新対象

IGCの入力は `output_IGP/Instagram_Logs`、`output_IGR/Instagram_Logs`、`output_IGS/Instagram_Logs` のみとする。既存 `output_IGC` と `output_IGX` を入力・補完元・正解にしてはならない。

IGCが更新できるのは次の6対象だけとする。

```text
output_IGC/Instagram_Logs/
├── Synapses/
│   ├── Tags/*.md
│   ├── Mentions/*.md
│   └── Locations/*.md
└── SystemLogs/
    ├── ハッシュタグ一覧.md
    ├── メンション一覧.md
    └── 場所一覧.md
```

`Posts`、`Reels`、`Stories`、`Salvage`、`media`、`timeline`、`RawData`、実際のObsidian Vault、iCloud配下はIGCの読み書き対象外とする。

#### 2.2.7 配置の禁止事項

* IGPのFeedメディアを全期間共通 `media/` へ移してはならない。
* IGR・IGS・IGXのメディアを半期内 `Instagram/media/` へ移してはならない。
* IGR・IGXを半期分割してはならない。
* IGP・IGR・IGS・IGXのRawData、Events、Timeline、状態管理ファイルを省略してはならない。
* 空の将来欄、既存のFrontmatter項目、人間が追記したSynapse本文を「現在使っていない」という理由で削除してはならない。

## 第3章 Markdown・本文仕様 (対応: [[03_データ構造・原本保持設計]] / [[04_データ取得・変換・出力設計]])

### 3.1 ファイルの文字コードおよび日付フォーマット
*   **文字コード**: 全てのMarkdownファイルは `UTF-8`（BOMなし）で保存する。
*   **日付表記**: 日付表示はすべて日本標準時（JST / UTC+09:00）基準とし、`YYYY-MM-DDTHH:MM:SS+09:00` のISO8601表記を強制する。

### 3.2 ノート本文テンプレート
YAMLヘッダーの下に、以下のフォーマットで本文およびメディア表示リンクを書き出す。

````markdown
---
(YAMLフロントマター)
---

[[instagram]]

(キャプション本文をそのまま改行を維持して記述)

![[(コピーされたメディアファイル名1)]]

![[(コピーされたメディアファイル名2)]]

---
[[instagram]] [[(ハッシュタグ1)]] [[@(メンション1)]] [[(位置情報名)]]
```
※最下部のリンク行は、`links` フィールドに格納されたWikiリンク配列をスペース区切りで一行で出力する。

## 第4章 Synapse・リンク生成仕様 (対応: [[03_データ構造・原本保持設計]] / [[04_データ取得・変換・出力設計]] / [[05_リンク・意味ネットワーク設計]])

### 4.0 v1.2におけるSynapse生成方針

v1.2では、ハッシュタグ・メンション・位置情報のいずれも、自動の揺らぎ判定および自動統合を行わない。

抽出されたタグ名・メンション名・場所情報は、個別仕様で生成対象としたものごとに、個別のタグメモ・メンションメモ・場所メモとして生成する。

将来、統合・分離・リンクON/OFFを行う場合は、Memory Synapse DBのObsidianプラグインまたは作業UIで、人間が確認して操作する。

個別のSynapseメモには、出現回数、投稿回数などの集計値を持たせない。関連する投稿メモへの接続は、各Synapseメモの `## 関連投稿` にWikiリンクとして保持する。

#### 4.0.1 メンションの抽出単位

各移行工程は、自分の入力データに含まれるメンション表記だけを扱う。

投稿本文、リール本文、ストーリー本文などの `@` 表記は、その場で実在アカウントへ確定せず、初期メンションメモへの入口として扱う。

フォロワー、フォロー中など本文外の関係データは、本文メンションと同一視せず、別の関係として保持する。後から同じアカウントかどうかを判断するのは人間の後工程とする。

ただし、`@` の直後に日本語の文章や店名が続いており、Instagramアカウント名として読むよりも、本文の一部または `#` の打ち間違いとして読む方が自然なものは、メンションとして抽出しない。

この場合、原文およびRawDataは変更せず保持し、必要なら後工程で人間が判断できるようにする。

#### 4.0.2 場所情報の抽出単位

場所情報は、投稿に付いたチェックイン1回を基本単位として扱う。

場所名、座標、住所はチェックインに付随する属性であり、自動統合の根拠にしない。

同じ場所名であっても、Instagram側のスポット、Facebook側で作成・選択されたスポット、住所登録、座標登録が異なる可能性があるため、同一場所として機械的に確定しない。

場所名一覧は確認用の索引であり、実在場所数または統合済み場所数として扱わない。

場所の統合・分離は、Obsidian上の後工程で人間が判断する。

### 4.1 タグおよびメンションの整形ルール
キャプションから抽出したハッシュタグ・メンションは、原文の表記をできるだけ保持したまま扱う。

ここで行うのは、意味を変えない最小限の整形だけであり、揺らぎ判定、自動統合、自動分離を意味しない。

1.  **記号除去**: 表示名とは別に内部処理で使う値を作る場合のみ、先頭の `#` や `@`（半角・全角）を除去してよい。
2.  **トリミング**: 前後の不要な空白だけを除去する。
3.  **表記保持**: 大文字小文字の違いは、人間が判断する前に同一視しない。
4.  **内部識別の分離**: ファイル名安全性が必要な場合は、人間向け名称とは別に変わらない内部識別子で扱う。

### 4.2 個別Synapseカードに保持する抽出事実とカテゴリ固有情報

IGP・IGR・IGS等の抽出工程は、自分の入力から確認できるハッシュタグ、メンション、位置情報の事実に加え、カテゴリ固有の初期情報、人間が後から育てるカテゴリ固有欄、関連投稿へのWikiリンクを持つ個別Synapseカードを、それぞれの `output_*` へ完成させる。人間が後から育てる欄は本仕様および個別仕様で定めた初期値まで設置し、未確認の値を推測して記入しない。

IGCは、各抽出工程から受け取った個別Synapseカードとカテゴリ固有欄を欠落なく保持して集約する。IGCがカテゴリ固有欄を作り直したり、空の仮カードを置き直したりしてはならない。複数カードの融合・分離・取消・復元・判断履歴を扱う大きな受け皿と共通作業台は、カテゴリ固有欄とは分けてIGC以降で用意する。

個別Synapseカードの完全なYAML、各項目の意味、初期値は、[[02_IGP移行仕様書v1.2#2.1.1 ハッシュタグメモ]]、[[02_IGP移行仕様書v1.2#2.1.2 メンションメモ]]、[[02_IGP移行仕様書v1.2#2.2.1 Location Synapseメモ]]を正とし、IGR・IGSも各個別仕様に従って同じ構造を出力する。

カテゴリ固有欄には、共通作業台の項目を重複して設置しない。融合、分離、取消、復元、判断履歴、操作履歴の具体的なYAMLは、本節の対象外とする。

#### 4.2.1 ハッシュタグの情報欄

ハッシュタグメモは、元のハッシュタグ表記と、人間が後から育てる自由メモ欄を `hashtag_note` として1件だけ持つ。その下の `## 関連投稿` に、そのハッシュタグが含まれる投稿メモへのWikiリンクを並べる。

````markdown
## 情報

```yaml
hashtag_note:                                 # ハッシュタグを人間が育てる欄
  hashtag: "#SampleTagA"                         # 投稿から抽出されたハッシュタグ表記
  note: null                                  # 自由メモ
```
````

````markdown
## 関連投稿

[[2024-01-06-12-00-00_IG_0006]]
[[2020-08-22-15-35-22_IG_1535]]
[[2020-08-22-16-08-30_IG_1534]]
````

`hashtag` は、先頭の `#`、大文字小文字、数字、記号を含め、投稿から抽出された表記を保持する。`note`（自由メモ）の初期値は `null` とする。

ハッシュタグメモには、投稿ごとの `hashtag_extraction`、`source_post`、`source_raw`、`raw` を繰り返し出力しない。`IGP_HASHTAG_EXTRACTION:(投稿ID):START` および `IGP_HASHTAG_EXTRACTION:(投稿ID):END` の生成管理印も出力しない。元のハッシュタグ表記と出典は投稿本文、投稿メモ、個別RawDataに保持し、ハッシュタグメモから各投稿への接続は `## 関連投稿` のWikiリンクで保持する。

#### 4.2.2 メンションの情報欄

メンションメモは、Instagramの `@文字列` と、人・店舗・団体等の情報を人間が後から育てる欄を `mention_note` として1件だけ持つ。その下の `## 関連投稿` に、そのメンションが含まれる投稿メモへのWikiリンクを並べる。

````markdown
## 情報

```yaml
mention_note:                                 # 人・店舗・団体などを育てる欄
  mention: "@sample_account_001"                   # 投稿から抽出された@表記
  name: null                                  # 実際の名前を人間が記入する欄
  phone: []                                   # 確認した電話番号を追加する欄
  web:                                        # メール・Webサイト・SNSなど
    - "https://www.instagram.com/sample_account_001/"
                                               # @表記から生成したInstagram URL
  note: null                                  # 自由メモ
```
````

````markdown
## 関連投稿

[[2022-05-11-17-20-55_IG_836]]
[[2022-05-24-16-48-12_IG_833]]
[[2022-05-26-18-03-07_IG_829]]
````

初期値と更新規則は次のとおりとする。

* `mention`（メンション）は、抽出事実の `raw`（元表記）と同じ `@文字列` を保持する。
* `name`（名前）は、人、店舗、団体等の実際の名前を人間が確認して記入する。初期値は `null` とする。
* `phone`（電話番号）は複数値とし、初期値は空配列 `[]` とする。既存の番号を上書きせず追加できるようにする。
* `web`（Web・連絡先）は複数値とし、メールアドレス、Webサイト、Instagram以外のSNS等を追加できるようにする。独立した `email`（メールアドレス）欄は設置しない。
* `web`（Web・連絡先）の初期値には、`mention`（メンション）の先頭の `@` だけを除き、`https://www.instagram.com/` の後ろへ付けたURLを1件入れる。ユーザーネーム中のピリオド、アンダースコア、大文字小文字は変更しない。
* 生成したInstagram URLは、元の `@文字列` に対応する参照先である。アカウントの現在の存在、現在の所有者、本人性を保証する値として扱わない。
* `note`（自由メモ）の初期値は `null` とする。
* `related_locations`（関連場所）は設置しない。同じ投稿に位置観測が存在することだけを理由に、人・店舗・団体等と場所の関係を確定しない。
* メンションメモには、投稿ごとの `mention_extraction`、`source_post`、`source_raw`、`raw` を繰り返し出力しない。
* `IGP_MENTION_EXTRACTION:(投稿ID):START` および `IGP_MENTION_EXTRACTION:(投稿ID):END` の生成管理印を出力しない。
* 元の `@文字列` と出典は投稿本文、投稿メモ、個別RawDataに保持し、メンションメモから各投稿への接続は `## 関連投稿` のWikiリンクで保持する。

#### 4.2.3 位置情報の情報欄

Location Synapseメモは、場所名、座標、住所、活動ID、活動の元ファイル、人間の記憶を一つの情報欄として持つ。その下の `## 関連投稿` に、その場所情報を持つ投稿メモへのWikiリンクを並べる。

IGPが出力する完全なYAML、各項目の意味、`null`、関連投稿、再実行時の保持規則は、[[02_IGP移行仕様書v1.2#2.2.1 Location Synapseメモ]] を正とする。共通仕様書には同じYAMLを重複記載しない。

Location Synapseメモには、投稿ごとの位置観測YAMLを繰り返し出力しない。元の場所名、チェックイン情報、出典は投稿メモと個別RawDataに保持し、Location Synapseメモから各投稿への接続は `## 関連投稿` のWikiリンクで保持する。

`IGP_LOCATION_OBSERVATION:(投稿ID):START` および `IGP_LOCATION_OBSERVATION:(投稿ID):END` の生成管理印を出力しない。

#### 4.2.4 再実行時の人間入力値の保持

既存の個別Synapseカードに人間または後工程が設定した値および追記した本文がある場合、抽出工程の再実行で削除または初期化してはならない。少なくとも、ハッシュタグの `hashtag_note.note`、メンションの `mention_note.name`、`mention_note.phone`、`mention_note.web`、`mention_note.note`、Locationの `activity_id`、`source_files`、`note` を保持する。原本から再確認する元表記、場所情報、関連投稿へのWikiリンクは、不足だけを補い、既存内容と原本が食い違う場合は自動上書きせず停止する。

## 第6章 エラー処理・例外処理仕様 (対応: [[04_データ取得・変換・出力設計]] / [[06_検証・運用・保全・拡張設計]])

### 6.1 エラー発生時の退避仕様
JSONのパースエラーや致命的なデータ破損が発生した個別エントリは、他の正常データの処理を妨げないよう、例外処理（`try-except`）で捕捉して退避処理を行う。
*   **保存先**: 該当期間フォルダの `SystemLogs/RawData/` への保存は行わず、原本JSONデータを `SystemLogs/RawData/` ではなくエラーログ記録に留め、対象の個別投稿処理のみをスキップする。
*   **エラーログ出力**: エラー情報（インデックス、エラー内容）を、第7章で規定するイベントログにカテゴリ `ERROR`、アクション `POST_PROCESSING_ERROR` として書き出す。
*   **完了扱いの禁止**: 1件でも技術的な処理失敗がある期間は完了扱いにしない。
*   **状態の分離**: 人間の判断待ちと技術的な処理失敗は別の状態として扱う。

## 第8章 状態管理・冪等性仕様 (対応: [[04_データ取得・変換・出力設計#4.10 再実行と不足修復]] / [[06_検証・運用・保全・拡張設計#6.6 再実行・修復・復旧]])

### 8.1 リラン制御（スキップ判定）
スクリプトが複数回実行されても、同じファイルの上書きや不要なメディアの再コピーを防止するため、以下の判定処理を実装する。
1.  **成果物対応確認**: 投稿メモの存在だけで処理済みと判断しない。RawData、メディア、Synapse、一覧など関連成果物との対応を確認する。
2.  **不足修復**: 正しい既存物は保持し、不足だけを補う。
3.  **食い違い時の停止**: 既存内容と元データが食い違う場合は自動で上書きせず停止して知らせる。
4.  **状態管理ファイル**: `Instagram_Logs/migration_state_*.json` には、関連成果物の対応確認まで完了した期間だけを保存する。

## 第10章 実データ監査結果に基づく事実差分 (新規追加章)

### 10.1 原本JSONスキーマの事実差分
* **通常投稿（Feed）用JSONの複数スキーマ混在**:

  * [差分反映: 既存の仕様書には構造Bのキーマッピング定義が存在しない事実を検出] (変更理由: 実データ監査結果に基づく事実)

* **Storiesデータ（stories.json）の構造**:
  トップレベルキー `ig_stories` を持つ辞書オブジェクトである。
  * [差分反映: 既存の仕様書にはトップレベルがリスト型ではなく辞書型である点の記述が存在しない事実を検出] (変更理由: 実データ監査結果に基づく事実)

### 10.2 メディア物理ファイルのパス不一致および一意性
* **URI相対パスの不一致**:

## 付録：改訂履歴

### A. v1.1改訂の背景と目的
当初定義された「仕様書v1.0」に基づくプログラムでのテスト移行にあたり、原本データ（JSONファイル構造および物理メディア配置）と仕様書の想定に不一致があることが懸念された。
処理中のメディア欠損やパースエラーを未然に防ぎ、移行処理の信頼性を100%にするため、実データ（原本JSONと物理ファイル計10,308件）を対象とした詳細な「実データ監査」が実施された。

監査の結果、以下のような**深刻な想定外の事実**が判明した：
1. フィード投稿用JSONに、キー構造が全く異なる別形式のJSON（`posts_1.json`）が混在している事実。
2. JSONに記録されたメディアの `uri` パスと、物理メディアの実際の配置場所（相対パス）が乖離している事実（全データの約25%）。
3. 仕様書に記載されている移行プログラムの接頭辞番号（`07_`等）と、実際のワークスペース内のフォルダ番号（`04_`等）が一致していない事実。

これらの乖離をそのままにして移行を実行した場合、大量のメディアリンク切れや投稿データパース失敗が不可避であった。そのため、実データから検出された「事実差分」のみを正確に反映し、安全かつ確実に移行を完了できるよう仕様をアップデートした「v1.1（実データ反映版）」を新規作成するに至った。

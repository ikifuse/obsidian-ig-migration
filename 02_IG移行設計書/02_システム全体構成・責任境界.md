# 第2章 システム全体構成・責任境界

## 2.1 全体構成の目的

本章は、プロジェクト全体を構成する各レイヤーと工程の役割を定義する。

本システムを一つの巨大なプログラムにせず、責任ごとに独立した工程を組み合わせる構造とする。

## 2.2 システム全体像

本システムは、次の責任で構成する。

- 原本となる外部データを保持する。
- データ種別ごとの独立した工程で、原本から派生データを生成する。
- 各工程の出力を、他工程や利用中データから分けて検証する。
- 検証した派生データを、人間の判断で知識基盤へ反映する。
- 投稿、時間、タグ、メンション、位置情報、Synapseの関係を知識構造として保持する。
- 人間が知識構造を確認し、意味判断を行う。
- AIは確定した知識構造をコンテキストとして利用する。

各責任は共有せず、入力、出力、変更可能範囲を分ける。

### 2.2.1 現在の完走対象

現在の完走対象は、IGP・IGR・IGSの抽出と検証、3系統だけを入力とするIGCの実行と`output_IGC`の検証、Memory Synapse DBのObsidianプラグインの動作、Obsidian上での成果確認である。Memory Synapse DBの成果確認には、人間判断による融合・分離、大きなカードの選択と変更、手書きによる成長、誤操作からの復旧を含む。

IGPはFeed、IGRはReel、IGSはStoryの元データを担当する。三工程は互いの不足を補う工程ではなく、それぞれが自分の入力から取得できる事実を完成させる。

IGP・IGR・IGSの完走はIGCへ渡すための中間地点であり、それだけをプロジェクトの現在の完走点にしない。

### 2.2.2 現在のIGC・プラグインと将来機能の境界

IGCは、IGP・IGR・IGSの抽出事実を出どころとカテゴリを失わずに集約し、Obsidianで確認する初期Synapse DBを`output_IGC`として組み立てる現在の工程である。IGXは入力に含めない。

Memory Synapse DBのObsidianプラグインは、`output_IGC`のSynapseを人間がObsidian上で確認し、現在許可された操作を行う入口である。現在の完走では、プラグインが動作し、成果をObsidian上で確認できることを検証する。

人間判断による意味融合、カテゴリ横断融合、個別分離、大きなカードの変更、取消・復元は、現在のMemory Synapse DBが担当する。AIによる意味の自動確定、未確定候補を長期間保存する常設作業台、将来情報源との融合は将来機能として分ける。既存の企画書、設計書、仕様書、コード、プラグインが存在するだけでは完成とせず、現在の完走条件に沿って実物を検証する。

### 2.2.3 現在の対象外

IGXは、通常分類で扱えない情報を将来救出する独立工程の候補であるが、現在の完走範囲とIGC入力範囲には含めない。現在の承認済み完走点へ到達した後に、実データと残存課題を確認して改めて判断する。

## 2.3 データソースの責任

### 2.3.1 外部データの役割

データソースは、本システムへ取り込む原本データを提供する。

現在の対象はInstagramである。将来Facebook、GPS、YouTube、資格学習、メモ、その他のライフログを追加する場合も、情報源ごとに独立した取込責任を持たせる。

### 2.3.2 原本の保持

データソースの原本は変更しない。処理には原本のコピーまたは原本から読み取った情報を使用し、再解析時は原本へ戻れる状態を維持する。

## 2.4 データ変換の責任

### 2.4.1 変換工程が行うこと

変換工程は、外部データを、人間とAIが確認できる派生データへ変換する。

JSON等の入力解析、日時情報の保持、Markdownとメタデータの生成、画像・動画の保持、文字情報の抽出等を担当する。投稿、RawData、メディアは、元のInstagram事実へ戻るための記録として維持する。具体的な項目、保存形式、処理順は仕様書で定義する。

ハッシュタグ、メンション、位置情報は、それぞれ別の抽出事実または観測事実として完成させ、元投稿と元データへ戻れる状態で後工程へ渡す。同じ投稿から同時に得られたことだけを理由に、それらが同じ意味または同じ存在を表すとは判断しない。

少なくともIGPは、ハッシュタグ、メンション、位置情報を、カテゴリ固有の情報、人間が後から育てるためのカテゴリ固有欄、関連投稿への接続を持つ個別Synapseカードとして`output_IGP`へ出力する。これらは空の仮カードではなく、IGPの入力から作成できる初期情報を持つ完成した工程成果物として扱う。

### 2.4.2 変換工程が判断しないこと

変換工程は、抽出した名前、表記、場所、関係の意味を確定しない。

意味が同じか、同じ人物や場所か、採用するか、統合するか等の判断を自動的に行わない。

変換工程は、複数カードの意味融合、カテゴリ横断融合、分離、取消、復元を判断せず、それらを扱う共通の融合関係と作業台を完成させない。

この責任境界は、承認済みの出力契約に従って、カテゴリ固有の自由メモ、連絡先、位置情報等を後から育てる欄を個別Synapseカードへ持たせることを禁止しない。個別カードを育てる欄と、Memory Synapse DBで人間が複数カードを融合して育てる大きなカードを区別する。IGCは、意味融合用の受け皿または作業台を作る工程ではない。

### 2.4.3 データ種別ごとの独立性

Feed、Reel、Storyは、それぞれ独立した変換責任を持つ。一つのデータ種別の失敗や変更が、他のデータ種別の処理、成果物、完了状態を破壊しない構造とする。

Salvageを将来設ける場合も独立工程とし、現在の三工程の責任や完成条件へ混ぜない。

## 2.5 出力と検証の境界

### 2.5.1 各工程の出力

各工程は、自分の入力から生成した成果物を、他工程の出力および利用中の知識基盤と分けて出力する。

個別工程の出力は、その工程の事実保持、欠落、対応関係、再実行性を検証するための単位である。

各抽出工程は、自分の入力から取得できる事実を、後工程が推測し直さなくてよい状態で出力する。後工程が存在することを理由に、元表記、元観測、出どころ、カテゴリ固有の情報を削らない。

`output_IGP`の個別Synapseカードにあるカテゴリ固有の情報、人間が育てる欄、元投稿へ戻るWikiリンクは、この工程の出力契約として保持する。IGCで完全一致カードを整理すること、またはMemory Synapse DBで大きなカードを育てることを理由に、これらを空欄だけの別形式へ戻したり、後工程で作り直す前提にしたりしない。

### 2.5.2 出力を使った実物確認

出力の存在だけで完成と判断しない。元データとの対応、必要な関連成果物、処理失敗、再実行時の保持を実物で確認する。

### 2.5.3 Obsidianへの部分的な確認投入との区別

一部の成果物をObsidianへ投入して実物を確認することと、工程全体またはプロジェクト全体が完成することを区別する。

## 2.6 知識基盤への反映境界

### 2.6.1 移行成果物と利用中データの区別

移行工程の成果物と、オーナーが利用する知識基盤を区別する。未検証の成果物を自動的に利用中データへ反映しない。

### 2.6.2 人間による反映判断

知識基盤への反映は、オーナーが実物と検証結果を確認した後に判断する。

IGC、Memory Synapse DB、Obsidianプラグインは現在の完走対象であるが、実行コードやプラグインが存在するだけでは完成としない。オーナーが`output_IGC`とObsidian上の実物を確認し、必要な検証が成立した後に完走判定する。

### 2.6.3 情報統合と実Vault投入を分ける

IGCが扱うSynapseとSystemLogsの情報統合と、Posts、Reels、Stories、RawData、media、Timelineを実際のVaultへ揃えて投入する手順は別の責任である。

一部の`output_IGP`を実Vaultへ投入して行った確認だけでは、現在の完走条件を満たさない。現在の完走では、IGP・IGR・IGSとIGCの成果物を確定し、`output_IGC`とプラグインの組合せをObsidian上で確認する。全成果物を日常利用Vaultへ正式反映する手順は、この確認と分けて設計する。

## 2.7 現在のIGC責任境界

### 2.7.1 IGCが担う責任

IGCは、各抽出工程が完成させたハッシュタグ、メンション、位置情報の事実を、入力系統と出どころを失わずに集約する。

IGCは、ハッシュタグ、メンション、位置情報のカテゴリを混ぜず、IGP・IGR・IGS間でカテゴリと人間が見る表記が完全に同じカードだけを、`output_IGC`の一件へ整理する。大文字小文字、記号、文字列の違いを正規化して同一扱いにしない。

完全一致カードでは、各入力カードが持つ関連投稿を`output_IGC`の一件へ集め、入力系統と元投稿へ戻れる関係を維持する。関連投稿以外の共通情報は、入力間で一致する場合に一組を保持する。一系統だけに存在するカードも失わず引き渡す。

IGCのカード集合は、IGP・IGR・IGSの件数を単純に足したものではなく、カテゴリと完全同一表記を単位とする集合である。SynapseカードとSystemLogsの一覧は、統合後の同じ集合と関連投稿を表し、相互に照合できる状態にする。

IGCは意味融合用の大きな受け皿、共通作業台、融合履歴を作らない。完全一致整理のためだけの融合グループID、所有者、融合元カードの付け替えも前提にしない。

### 2.7.2 IGCが行わないこと

IGCは、名前の近さ、ファイル名、表記揺れ、座標、住所、投稿内の共起だけを理由に意味の同一性を確定しない。同一カテゴリ内の意味融合とカテゴリ横断融合を実行しない。`#ABC`と`#abc`等、表記が異なるカードも別々に保持する。

カテゴリと表記が完全に同じカード間で、関連投稿以外の共通情報が異なる場合、どれか一つを正解として選択、上書き、推測補完しない。入力を失わず不一致を検出できる状態にし、停止・報告方法は仕様書で定義する。SynapseカードとSystemLogsの一覧が対応しない場合も、片方だけを正解として自動修復しない。

現在のIGC入力はIGP・IGR・IGSの3系統とし、IGXを含めない。具体的な入力単位、識別、集約、ファイル構造、置換方法は仕様書で定義し、現行IGCコードと成果物を照合して検証する。現行コードを存在するだけで正解または完成と固定しない。

## 2.8 Memory Synapse DBの責任境界

### 2.8.1 操作設計を確認するブラウザー環境

ブラウザー環境は、大きなカード、受け皿、融合、分離、手書き、取消の見え方と操作を、Obsidianプラグインへ固定する前にオーナーが確認する設計検証環境とする。

ブラウザー環境で確認した画面や操作を、そのまま完成仕様または本番成果物とは扱わない。検証用データまたは安全な複製を用い、日常利用データを直接変更しない。確認結果を設計へ反映し、オーナーの設計承認後に仕様へ渡す。

### 2.8.2 Obsidianプラグインの責任

Memory Synapse DBのObsidianプラグインは、人間が`output_IGC`のSynapseと出どころを確認し、Markdownを実体として知識を育てる最終操作入口とする。

プラグインは、個別カードの融合・追加・分離、大きなカードの選択と変更、大きなカード内の手書き情報の追記・編集、操作前の影響確認、取消・復旧を扱う。抽出工程で欠落したInstagram事実を推測して補わず、似ているという理由だけで意味を自動確定しない。

具体的な画面配置、ボタン、入力欄、ドラッグ＆ドロップ、確認文言、使用言語、保存手順は仕様書で定める。

### 2.8.3 現在機能と将来機能の境界

現在は、人間が同じ知識として育てたいと判断したTag、Mention、Location等のカードを、カテゴリや枚数に制限されず一つの大きなカードとして扱い、必要な一枚だけを分離できる構造を対象とする。

AIによる意味の自動確定、未確定候補を長期間保存・比較する常設作業台、将来情報源の取込と融合は現在の責任へ含めない。操作履歴と復旧情報は、未確定候補を置く作業台とは別の保全責任として扱う。

## 2.9 意味ネットワークの責任

### 2.9.1 関係情報の役割

意味ネットワークは、投稿を孤立したファイルとして保存するのではなく、タグ、メンション、位置情報、時間、Synapse等との関係を保持する。

### 2.9.2 Synapseの役割

Synapseは、個々の投稿や観測から、後で人間が意味を確認し、知識を育てるための接続先である。

Synapseの存在と、人間が複数の情報を同じ意味または同じ実体と判断した状態を同一視しない。

## 2.10 人間判断の責任

人間は、採用、非採用、名寄せ、意味統合、分離、代表名、意味に関わる変更を判断する。

将来の操作画面は、人間が判断するための入口であり、意味を自動決定する主体ではない。

## 2.11 AI利用の責任と禁止範囲

AIは、確定した知識構造の検索、要約、分析、推論、提案、開発補助に利用する。

AIは原本の事実、情報同士の意味、設計変更を単独で確定しない。利用するAIが変わっても、原本と知識構造を変更せず利用できる状態を維持する。

## 2.12 レイヤー間の関係

### 2.12.1 情報の流れ

原本から派生データを生成し、派生データを検証し、人間の確認を経て知識基盤へ反映する。

これは責任の流れを示す設計であり、具体的なプログラムの実行順を定めるものではない。

### 2.12.2 元データへ戻れる構造

派生データと知識構造から、元の入力、元の表記、関連する投稿やメディアへ戻れる関係を維持する。

### 2.12.3 人間判断を経由する変更

意味ネットワークに影響する変更は、人間の判断を経由する。意味判断の結果に合わせて原本を書き換えない。

## 2.13 全体構成の成立条件

- 各工程の入力、出力、変更可能範囲を区別できる。
- 一つの工程の失敗や変更が、他工程の正しい成果物を破壊しない。
- 原本、派生データ、人間の意味判断を区別できる。
- 個別出力の確認と、知識基盤への反映と、プロジェクト全体の完成を混同しない。
- 未承認の将来機能を、現在の完成条件に追加しない。
- 抽出工程、IGC、Memory Synapse DBの責任を重複させない。
- `output_IGP`等の個別Synapseカードにあるカテゴリ固有情報と育てる欄を、IGCの完全一致整理またはMemory Synapse DBの大きなカードと混同しない。
- 同名だが内容と出どころが異なる観測を、後工程で一つへ潰さない。
- IGCは完全一致整理に限定し、Memory Synapse DBの意味融合、分離、判断履歴を先取りしない。
- Memory Synapse DBの意味融合でも元カードを失わせず、Wikiリンクを中心に人間が関係を確認できる。
- 融合管理だけを目的とするカードを自動生成せず、人間が選んだ既存カードを育てる。
- ブラウザーで確認した設計を、承認なしにObsidianプラグインの仕様または実装へ固定しない。
- IGCとObsidianプラグインは現在の完走対象として実物検証し、未検証の具体方式や将来機能を完成済みとして扱わない。

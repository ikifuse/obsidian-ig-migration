# 第5章 リンク・意味ネットワーク設計

## 5.1 リンクを作る目的

リンクはGraph Viewの見た目を作るためではなく、投稿と、時間、表記、場所、人物、将来の意味との関係を、人間とAIがたどれるようにするために生成する。

リンク生成は関係の入口を作る処理であり、関係の意味を確定する処理ではない。

## 5.2 リンク対象の分類

Instagram移行では、少なくとも次の関係を区別する。

- ハッシュタグとの関係
- メンションとの関係
- 位置情報との関係
- Timelineによる時間的な関係
- Synapseを介した意味上の関係

異なる種類の関係を、一つの共通リンクとして意味を失う形にまとめない。

## 5.3 Timelineによる時系列関係

投稿間の時間的な関係は、個々の投稿を破壊的に書き換え続ける方法ではなく、期間ごとの時系列情報から確認できる構造とする。

前後の投稿を各投稿へ固定して持たせず、将来ほかの情報源を追加しても、既存投稿の書き換えを必要としない設計とする。

具体的な期間単位、並び順、ファイル名、出力形式は仕様書で定義する。

## 5.4 ハッシュタグとの関係

投稿本文から取得したハッシュタグ表記と、元投稿との関係を保持する。

人間が確認する前に、表記違いを同じ意味として自動統合しない。投稿とハッシュタグの関係を生成することと、ハッシュタグの意味を確定することを区別する。

リンク先を作るために元表記の大文字小文字や記号を失わせない。ファイルシステム上の安全なリンク先と、人間が見る元表記を必要に応じて分離する。IGCにおける具体的なリンク先と表示方法はIGC仕様書で定義する。Memory Synapse DBにおける融合関係の設計条件は本章で定め、具体的な識別方法は仕様書で定義する。

抽出条件、重複の扱い、リンク形式、Synapseの出力形式は仕様書で定義する。

## 5.5 メンションとの関係

投稿本文から取得したメンション表記と、元投稿との関係を保持する。

投稿本文にある表記だけから、実在アカウント、現在の所有者、人物や店舗との同一性を自動確定しない。

同じ投稿に位置観測が存在しても、それだけでメンション対象の人物、店舗、団体と場所の関係を生成しない。投稿本文への登場、フォロー関係、人間が確認した連絡先を別の関係として保持する。

抽出条件、リンク形式、Synapseの出力形式は仕様書で定義する。

## 5.6 位置情報との関係

投稿に付随する位置情報と、元投稿との関係を保持する。

同じ場所名、近い座標、似た住所であることだけを理由に、複数の記録を同じ場所として自動統合しない。

物理的な位置情報と、オーナーが後から加える意味や記憶を区別する。

同じ名前のLocation接続先へ複数の位置観測を集める場合も、各観測と元投稿への関係を残す。一つの代表座標または最初の観測だけを残して、他の観測を関連投稿リンクだけへ縮めない。

## 5.7 Synapseの役割

Synapseは、投稿から抽出した関係情報を確認し、人間が後から意味、関係、メモを育てるための接続先である。

自動生成されたSynapseが存在することと、人間が意味や同一性を確認したことを同じ状態として扱わない。

### 5.7.1 個別Synapseカードと初期Synapse DB

少なくともIGPは、ハッシュタグ、メンション、位置情報を、カテゴリ固有の初期情報、人間が後から育てるカテゴリ固有欄、関連投稿へのWikiリンクを持つ個別Synapseカードとして`output_IGP`へ出力する。個別Synapseカードは空の仮カードではなく、生成直後から一件の情報として確認できる工程成果物である。

現在のIGCが組み立てる初期Synapse DBは、少なくとも次を相互に照合できる構造とする。

- 抽出工程から受け取った元表記、観測、出どころ、カテゴリ固有情報
- 同一カテゴリ・完全同一表記を単位に整理したSynapseカード
- 完全一致カードに属するIGP・IGR・IGSの関連投稿
- 統合後のSynapseカード集合と一致するSystemLogsの一覧

IGCは、意味融合用の大きな受け皿、共通作業台、融合・分離の判断欄を初期Synapse DBへ追加しない。具体的なMarkdown項目、リンクの保存位置、完全一致条件、競合時の停止・報告、検証条件はIGC仕様書で定義し、現行コードと`output_IGC`の実物で検証する。

### 5.7.2 Wikiリンク型の成長したSynapse

成長したSynapseは、人間が複数の事実やSynapseを同じ知識へ関係付けた状態である。自動生成されたファイル、同名の一覧、空の入力フォームだけを、成長したSynapseと呼ばない。

一つのSynapseが、Tag、Mention、Location等の複数の性質を持つ場合も、どれか一つのカテゴリへ変換して他の専門情報を失わせない。

現在のMemory Synapse DBでは、ハッシュタグ、メンション、位置情報等のカテゴリを越えて、人間が同じ知識だと判断したカードを必要な枚数だけ一つの大きなカードとして育てる。

融合時も、個別カードの種類、カテゴリ固有情報、関連投稿、元データへのつながりを維持する。人間が選んだ既存の個別カードを大きなカードとし、そのカードから他の融合構成要素へのWikiリンクを保持する。融合用の新しいカードを自動生成しない。

大きなカード自身も受け皿の構成要素として扱うが、自分自身へのWikiリンクを必須にしない。大きなカード自身の情報と、他の個別カードへの融合リンクを合わせて受け皿を構成する。

## 5.8 観測事実と意味判断の区別

元投稿に表記、チェックイン、日時、関係が存在したことは観測事実として保持する。

複数の観測が同じ人物、場所、出来事、概念を表すかどうかは、観測事実とは別の人間判断として扱う。

人間が後から意味を変更しても、元の観測事実へ戻れる状態を維持する。

## 5.9 同名・同表記・同一投稿内共起の扱い

同じ名前または同じ表記であることは、抽出結果を探すための手がかりであり、意味が同じであるとの確定ではない。

同名の入力が異なる観測、座標、住所、元投稿、カテゴリ固有情報を持つ場合、どれか一つを選択して他を捨てない。同じ名前のもとに複数の観測が存在する正常な状態として、各出どころを保持する。

同じ投稿内に複数種類の情報が存在することは共起であり、それだけで人物と場所、タグと場所、タグとメンション等の意味関係を生成しない。

## 5.10 人間が判断する範囲

採用、非採用、名寄せ、同じ意味であるとの判断、統合、分離、代表名、リンクの意味変更は、人間が確認して決定する。

ブラウザー上の設計検証環境とObsidianプラグインは、この判断を行うための操作入口であり、意味を自動決定する主体ではない。

### 5.10.1 大きなカードの選択

融合時は、融合対象に含まれる個別カードから大きなカードの候補を提示し、人間が最終選択する。候補提示では、情報の役割が具体的なものを中心にしやすくするため、Mention、Location、Tagの順を推奨順位とする。同じ種類が複数ある場合は、ドロップまたは融合操作を受ける側を初期候補とする。

推奨順位と操作方向は初期候補を示すために使い、人間の選択を強制しない。融合対象、大きなカード候補、融合後の構成を実行前に確認できることを成立条件とする。

現在の大きなカードを分離する場合も、残った個別カードから次の大きなカード候補を提示し、人間が選ぶ。残りが一枚の場合は候補を選ばせず、融合状態を終了する。

## 5.11 三種類の整理・融合

意味ネットワークの整理と融合では、次の三種類を区別する。

1. 入力系統間の完全一致整理：同一カテゴリで、人間が見る表記と関連投稿以外の共通情報が完全に同じ生成結果を、出どころを残して一件へ整理し、関連投稿を集める。
2. 同一カテゴリ内の意味融合：表記が異なるが同じ意味または同じ実体だと人間が判断したSynapseを融合する。
3. カテゴリ横断融合：Tag、Mention、Location等が異なっても、同じ存在だと人間が判断したSynapseを融合する。

一番目は現在のIGCが扱う責任である。大文字小文字、記号、文字列が異なる表記は別カードとして残す。何を完全一致と判定し、関連投稿以外の内容差がある場合にどう停止・報告するかはIGC仕様書で定義し、現行コードと`output_IGC`で検証する。内容差を無視して一つを選択または上書きしない。

二番目と三番目は、現在のMemory Synapse DBで人間が確認後に行う責任であり、IGC責任ではない。一つの大きなカードには両方を含め、カード数またはカテゴリ数に設計上の固定上限を設けない。具体的なWikiリンクの記法、保存位置、同名カードの指定方法は仕様書で定義する。

## 5.12 Memory Synapse DBの可逆性条件

Memory Synapse DBは、人間が複数のSynapseを融合した後も各個別カードを残し、誤って融合したカードだけを分離でき、残りのカードと大きなカードを維持できることを成立させる。

個別カード固有の情報は個別カードへ残す。大きなカードが通常表示の場合は、その個別カードの主要情報を大きなカード上部へ表示し、受け皿では重複を避けた短縮表示にする。短縮表示によって元情報を変更または削除しない。

人間が大きなカード内の手書き機能を使う場合は、新しいカードを作らず、現在の大きなカードに人間の意味情報を追加する。手書き情報を大きなカード上部へ表示し、以前上部に使われていた個別カードを受け皿の通常表示へ戻す。

一枚を分離しても、他の個別カードと大きなカードの手書き情報を維持する。手書き情報の一部を分離先へ移す場合、または大きなカードの交代・解体によって保持先を変える場合は、人間が対象と保持先を確認し、プラグインが文章の所属を推測して自動分配しない。

融合関係の具体的な保存位置、判断履歴の形式、取消単位、一括保存、途中失敗の検証と復旧、UI操作は仕様書で定義する。

## 5.13 Wikiリンクによる識別と変更追従

個別Synapseカードは、元投稿等へ戻るWikiリンクを維持する。IGCの完全一致整理では、カード融合用の抽出ID、融合後カードID、融合グループIDを追加しない。これは、同じLocation内の複数観測を区別する識別を廃止するものではない。

Memory Synapse DBでもWikiリンクを融合接続の中心にする。融合関係と、一般的な関連・出典・関連投稿のWikiリンクを区別できる構造とする。一つの個別カードは同時に複数の大きなカードへ融合させず、複数の知識との一般的な関連は融合以外のリンクで表す。

専用IDは前提にしない。同名メモの区別、改名・移動への追従、未解決リンクや意図しない別カードへの接続の検出に必要な具体規則は、Obsidianの実物を確認して仕様書で定義する。

## 5.14 自動統合・自動意味付けの境界

現在の移行工程では、キャプション、タグ、共起、名前、座標等から、テーマや同一性を自動確定しない。

自動トピック生成、自動クラスタリング、自動的な意味統合・分離を、移行完了の条件にしない。

## 5.15 将来の意味ネットワーク拡張

知識構造は、投稿等の観測記録、タグ・人物・場所等の中間的な接続先、人間が確認した出来事や概念等を関係付けられる構造を目指す。

将来の抽象的なノードや分析機能は、原本と既存の関係を壊さず追加できるようにする。未検証の自動分析、Graph最適化、再構築方式を現在の確定設計にはしない。

現在のIGCに必要なMarkdown項目、完全一致条件、関連投稿の集約、SystemLogsとの対応、Wikiリンク、競合時の停止・報告はIGC仕様書へ具体化し、現行コードと実成果物を事実として照合する。既存資料やコードに存在するだけで正解または完成とせず、企画書、確定した責任境界、実データによるサンドバッグ検証を経て採否を決める。

Memory Synapse DBの融合関係、大きなカード、受け皿、手書き情報、分離、取消・復元は、本章と対応分冊の設計条件をMemory Synapse DB仕様書へ渡す。具体的なリンク記法、画面操作、確認文言、保存処理、エラー処理は、ブラウザーで設計を確認した後に仕様工程で確定する。
